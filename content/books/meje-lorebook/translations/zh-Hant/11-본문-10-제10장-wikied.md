# 第10章　Phase 0～3——從協議到試點部的前半程

設想一場電影的第一次編劇室會議。編劇們圍坐的桌上放著一份列印PDF，封面寫著 *NDA* 與 *Show Bible v1.0*。會議的第一個小時用來共同確認各頁包含什麼；此後，整季的正式工作才能開始。LOREBOOK 專案的第一個月與此相似。它不是撰寫正文的時間，而是把一個 IP 的所有決定彙集到同一份文件中的時間。

Phase 0 的產物不是一本書，而是一組協議。

最初2至4周不寫正文，而是把如何處理1,500個詞條的決定彙整合約10至20頁的文件。它會成為隨後9至14個月工作的 SSOT；只有先整理好它，骨架、試點部與擴充套件才能依次展開。

*Phase* 一詞借自遊戲開發現場，常用於前一階段產物成為後一階段輸入、依賴關係清楚的工作。LOREBOOK 分為Phase 0至6七個階段，每一階段的驗收線就是進入下一階段的門檻。本章討論前四個：協議（0）、骨架（1）、試點部（2）與擴充套件（3），即最初約4至6個月。

## 七個Phase的時間流

Phase 0 用2至4周製作協議。Phase 1 是1至2周的骨架構建，自動生成1,500個空的 LOREBOOK 檔案並轉換 frontmatter。Phase 2 用3至4周製作一個試點部並由主審驗收。最長的Phase 3用4至6個月撰寫其餘11部。Phase 4用1個月驗證一致性，Phase 5用1個月製作附錄，Phase 6用2個月轉換出版版與網頁版。

總計9至14個月。純手工作業需要18至24個月；與 AI 工作夥伴協作，可縮短到接近一半。

每個詞條用六種標記追蹤：☐未寫，◐僅後設資料，◑散文初稿，◕散文完成，●含交叉引用的完成稿，✓校對與驗收完成。一個`_進度狀態.md`檔案每週更新1,500個詞條的狀態，Phase門檻也據此判斷。

本章講0、1、2、3；下一章講4、5、6。

## Phase 0——用一個月撰寫協議

Phase 0需要2至4周，是因為所有決定並非權重相同，彼此又有依賴關係；前一項未完成，就不能進入下一項。

首先要做的是**輸入確認**。檢查Vault的1,500個詞條、13列frontmatter、九分類與世界觀軸排列，以及Obsidian圖譜檢視是否正常。輸入不足時，就在Phase 0內返回Librarying補強。13列有缺項，回到第二次整合；九分類或世界觀軸未定，回到第一次迴圈。輸入整理完畢，才能確定部結構與語調指南。

權重最大的兩項決定是**部結構**與**語調指南**。部結構運用第4章程式，從10部、12部、14部型中選擇，並依次檢查IP規模、世界觀軸分佈、型別行動者、維度多層性與短篇資產。語調指南把第7章的七種聲音對映到各部，並按IP性質調整比例，另存為約2至3頁的Markdown供所有部參考。兩者決定書的**身份**，也是Phase 2試點部的驗收依據。

下一組是三項政策。**SSOT層級**規定資料衝突時的優先順序，標準為角色表、主審確認的短篇、LOREBOOK、Vault散文、規則書；某些IP會把規則書上移，外語原作也可能居於最高位。**短篇分離政策**按第9章在另冊出版（推薦）、併入第11部、引用框三種方案中選擇。**LOREBOOK型別**從第2章五種變體A至E中選擇。多數IP採用標準百科型A，若採用其他型別，則在協議中明確差異。這三項決定書的**政策**。

最後是兩項實務決定。**出版規格**先確定標準B5開本、約450頁、插圖預算、裝訂、封面加工與豪華版選項的大方向，再在出版階段精調。**日程與預算協議**決定9至14個月內各Phase如何分配時間。即使不同IP有所調整，也必須把整體安排寫入協議，以便專案中途檢查。

八項決定合成約10至20頁的Markdown文件，稱為**Bible**。Phase 1至6的一切工作均以它為依據，出現問題也總要返回Bible。若試點部審查發現語調偏差，就修改語調指南；若Phase 3暴露部結構不合理，就重新作出結構決定。

這裡還要引入**主審**。每個IP設一人，作為本LOREBOOK的最終決策者，負責部結構、驗收、出版規格與每一Phase的GO/NO-GO門檻。個人專案由製作者兼任，位置近似電視劇的總製作人或遊戲工作室總監。

## Phase 1——構建空骨架的時間

Phase 1幾乎完全自動化。1至2周內構建空的LOREBOOK骨架，人主要負責驗收。

七步流程大多是機械操作。先建立作為Obsidian Vault的根目錄`[專案 LOREBOOK]/`，在`_Spec.md`中整理Bible的關鍵決定。再建立從`00 序言`到`12 附錄`的12個部目錄及各自空的`README.md`，隨後建立每部的章目錄與空的章README。

核心自動化是建立1,500個空詞條Markdown。每個檔案的frontmatter由Librarying的13列自動轉換，正文保持空白。把13列CSV交給AI工作夥伴，約5分鐘即可生成1,500個檔案。最後建立根README，並用1,500個☐初始化`_進度狀態.md`。

人工部分是檢查frontmatter轉換：約30分鐘隨機開啟50個樣本，確認結果準確。

產物是空的LOREBOOK骨架。Obsidian側欄排列著1,500個Markdown檔案，圖譜檢視顯示由wikilink連線的節點，但節點內部尚無正文。Phase 2至3將逐一填入內容。

## Phase 2——一部決定整本書的命運

Phase 2是專案最大的分岔點。完成一個試點部並由主審驗收。GO則進入Phase 3；NO-GO則更新Bible後重寫試點。

若帶著缺陷進入Phase 3，同一缺陷會擴散到12部，直到Phase 4一致性驗證時才暴露；此時所有資料已經寫完，恢復成本極高。反之，試點部若準確跨過驗收線，其語調、SSOT引用和對絕對禁項的規避會自然延伸到其餘各部。一部的決定定義全書格式。

試點通常選第2部“世界法則”或第4部“地理”，以最能顯現IP身份者為準。虛構IP“○○”適合第4部，因為巫俗、料理與日常的混合最清晰地體現在城市、街區和生活景觀中。若IP由SF或奇幻的魔法機制定義，則第2部更合適。D&D《Forgotten Realms》這類大型IP以魔法學派作為新設定的先導，也是因為機制決定IP性質。

試點流程有六步：選擇試點部；撰寫部README與全部章README；先把連線最多的3至5個hub詞條寫成散文；按第7至9章驗證語調、結構和SSOT引用模式並反饋到語調指南；進行部級一致性檢查，查詢死鏈與五項絕對禁項；最後由主審作出Phase 3的GO/NO-GO決定。

六步約需3至4周，其中hub散文化與主審驗收耗時最多。

是否合格看五項：部README能否作為指南；全部章README是否完成；核心3至5個詞條是否達到◕以上；語調、結構、SSOT引用是否符合第7至9章；五項絕對禁項是否為0。五項全✓即GO，任何一項未達即NO-GO。

NO-GO時分析原因來自Bible還是執行。若來自Bible，更新後重寫；若來自執行，則區域性或整體重寫。一部的決定左右整個專案，不能草率處理。

也可以不做試點而同時推進12部，但不推薦，因為相同缺陷會鋪滿全書並延續到Phase 4，難以恢復。

電影製作中的**試播集**承擔同樣功能。它在整季開始前確定語調、人物與導演方式。《Lost》試播整合本約1,400萬美元，接近普通單集的五倍；《Game of Thrones》試播集曾整集重拍，原因都在於一集會決定之後6至8季。LOREBOOK試點部也先檢驗IP的語調、結構與一致性，再讓其餘11部沿同一路徑推進。

## Phase 3——4至6個月的正式寫作

Phase 3是最長階段，撰寫試點之外的11部。每部約2至3周，總計4至6個月。

順序不是隨機的，而是按書的身份逐漸穩定來安排。首先寫第1、2、3部（世界語調、世界法則、歷史與年表），先確定身份。隨後寫第5、6、4部（社會、日常、地理），填充體量最大且穩定的生活骨架。再寫第7、9、10部（行動者、圖鑑），在人物與勢力SSOT穩定後整理為卡片；卡片較輕，即使前文修改也容易恢復。最後寫第8部（維度與空間圖鑑），因為它最具挑戰，必須吸收其他部的一致性。

順序即1 → 2 → 3 → 5 → 6 → 4 → 7 → 9 → 10 → 8。不同IP可稍作調整，但大方向是**身份 → 日常骨架 → 卡片化 → 整合**。

每部重複試點流程：撰寫部README與章README；從hub詞條開始散文化；撰寫標準詞條與卡片詞條；進行部級一致性檢查；按實際篇幅與狀態更新部README。五步共重複11次。

插圖委託同步進行。一部達到◕後才開始委託。正文未定就先委託，正文更新時必須重新下單，恢復成本很大，因此標準原則是**正文v_確定後委託**。

建議每週更新`_進度狀態.md`。某部延遲時能立刻發現，以每週的小檢查防止4至6個月的長專案發生大偏移。

典型的五個月安排如下：第1週末，第1部5個hub達到◑；第2周增加5個標準詞條；第3周卡片達到◕，第1部過線。第4周開始第2部，同時外包第1部7至10幅插圖。若第2部發現語調不符，當場決定重寫。第8周第2部過線，第12周第3部，第16周第5部，第20周第6部，第23周加入第4部的綜合概覽詞條。每週固定約一小時檢查，可避免重大事故。

產物是所有部達到●至✓的LOREBOOK正文。1,500個詞條全部有散文與wikilink，插圖委託完成80%以上，Obsidian圖譜顯示1,500個節點與5,000至10,000條連結。但此時仍未最終完成；Phase 4至6還要依次處理一致性驗證、附錄與網頁/出版封裝。

## 每個Phase的進入門檻

明確每一Phase的驗收線，構成專案的秩序。

從Phase 0進入1，須完成輸入確認與Bible，並由主審檢查。從1進入2，自動結構檢查驗證目錄骨架、Spec、總README與全專案為☐或◐的`_進度狀態.md`。從2進入3的門檻最嚴格：試點部達到◕至●，語調指南已更新，主審下達GO。從3進入4，則要求所有部全部詞條達到◕以上且插圖委託已啟動，自動檢查與主審驗收共同進行。

未透過就返回當前Phase。不建議臨時繞行，因為不足會擴散到下一階段，令恢復成本劇增。因此門檻必須執行。

## 虛構IP“○○”的Phase 0～3案例

Phase 0約3周。輸入確認發現Vault已有1,200個詞條。因有60名角色，採用把第9、10部分開的12部型；七種聲音按部對映，並提高巫俗領域的圖鑑體與現場體比重。SSOT順序為角色表 → 短篇 → LOREBOOK → Vault → 規則書。30篇短篇另冊出版，型別採用標準百科型A。出版規格為B5精裝約450頁、插圖預算1,500萬韓元，Phase 0至6總計11個月。產物是一份18頁Bible。

Phase 1約1.5周。建立`[○○_LOREBOOK]/`、`_Spec.md`、12個部目錄、每部3至6章、1,200個已自動轉換frontmatter的空詞條、總README，以及全詞條為◐的`_進度狀態.md`。產物為空骨架。

Phase 2約4周。選擇第4部“地理”作試點，因為巫俗、料理與日常混合的身份在城市和街區最鮮明。完成第4部README與5個章README後，把四個hub（六風格地區概覽、梅里屯、海原餐飲街、舊巫堂街）寫成散文。語調、結構與SSOT引用檢查確認死鏈0、絕對禁項0。主審指出第四個詞條部分語調偏差，區域性重寫後獲得GO。產物是第4部達到◕至●。

Phase 3約5個月，順序為第1部（3周）→ 2（4周）→ 3（4周）→ 5（3周）→ 6（3周）→ 7（3周）→ 9（4周）→ 10（3周）→ 8（3周）。每部遵循README → hub → 標準詞條 → 卡片詞條 → 一致性檢查 → README更新。第1部達到◕時開始委託插圖，五個月內完成90%；每週五更新`_進度狀態.md`。產物是所有部達到●至✓。

至此LOREBOOK正文全部寫完。下一步是驗證一致性、製作附錄，並轉換為出版版與公開網頁版。
