# 第13章　一致性、版本與插畫——活著的 LOREBOOK

從 1987 年第一版到 2024 年第五版，D&D《被遺忘的國度》在約三十七年間經歷了五次重大修訂。期間約有兩百本短篇集、小說和戰役模組出版，每一次都有新人物、新城市、新事件積累到 LOREBOOK 中。費倫從 1372 DR 走到 1492 DR；原本是一座城市的扎卡拉變成新興勢力的根據地；法師伊爾明斯特的一生由新作者重新講述。這三十七年的流動正是 LOREBOOK **活著**的範本。一個 IP 的資料不會在出版後靜止，而會延續為每季度、每年的小更新，以及每五年、十年的大型新版本。

v1.0 出版後，每季度約兩三天的定期更新使 LOREBOOK 保持活力。

IP 運營期間會有新短篇、角色更新、Vault 增量更新、多語言版追加、詞條更名。每次變化，LOREBOOK 都隨之跟進。每季度一次的小更新與每五至十年一次的大型新版本構成運營的兩層。若第一次週期把書的**材料**變成了**書**，運營階段則讓這本書成為**跟隨時間的資料**。

本章討論這種運營：一致性驗證、版本管理、插畫委託、AI 插畫政策、更新週期，以及五年運營後的面貌。它從第一次週期結束之處開始，一直看到運營紮根的階段。

## 一致性驗證——每季度的小檢查

運營的第一部分是一致性驗證。第 11 章將其作為 Phase 4 處理；在運營階段，它會在每季度、新短篇出版後、角色卡更新後定期執行。

它由三個階段組成。第一階段是**自動檢查**。審校指令碼在首次檢查中找出失效連結、重複詞條、frontmatter 一致性、絕對禁止項和名詞 grep。約需一至兩小時；每季度一次、新短篇出版後一週內、新賽季或大型 IP 更新後一月內、角色卡 v_最終更新後一月內執行。

第二階段是**人工抽樣審校**。自動檢查透過後，由人直接檢視 1,500 個詞條中的三十至五十個隨機樣本，修正自動檢查無法發現的細微問題。它約需一至兩天，並與自動檢查同時進行。樣本應按部均勻分配——每部約三至五個——遵循 hub、標準、卡片型 1:2:7 的比例，並優先審校最近更新的詞條。

最後是**整合構建審校**。這一階段負擔較大，不在每季度執行，只在第一次週期出版前、v2.0 等新版本出版前，以及季節轉換、世界觀重大變動等大型 IP 更新後進行。由人構建整個 LOREBOOK，將其作為一本書通讀，並對是否達到合格線作出最終判斷。

三個階段結合起來，形成防止運營中資料偏移的多層檢查體系：自動檢查每季度先行抓取問題，人工樣本補足細微部分，整合構建在大轉折點檢視全書。

## 與 Vault 同步

LOREBOOK 運營的核心是與 Vault 同步。Librarying 階段的產出是 LOREBOOK 的輸入，因此 Vault 更新時，LOREBOOK 也應同步更新。它分為兩種形式。

第一種是**從 Vault 到 LOREBOOK 的批次更新**。Vault 中某個詞條改名或新增詞條時，LOREBOOK 接收該變化並更新。

以虛構 IP“○○”為例。假設 Librarying 第二次整合把一個詞條從“黑孔”改為“暗淵”。Librarying 第三次重建生成新的 `.md`，舊 `.md` 以 `version=廢止` 隔離。將 LOREBOOK 正文和編輯筆記中的舊術語批次替換為新術語，可使用 `sed` 自動化或 Obsidian 批次修改。將舊術語加入 LOREBOOK frontmatter 的 `aliases`，讀者以舊寫法搜尋時仍能找到新詞條。最後在 `_檢查清單.md` 記錄：正文舊術語殘留 0 項、aliases 登記 100%、附錄一術語表已記錄變更歷史。整個流程約一至兩小時完成。

第二種是**新術語的雙向反饋**。LOREBOOK 工作中可能出現 Vault 尚未收錄的新術語，例如新短篇自然引入的新詞。可在 LOREBOOK 詞條編輯筆記中標記“Vault 未登記：[術語名]”；在下一季度 Librarying 的第一、第二次增量週期中，將該術語登記到九分類之一併填寫十三欄位。第三次骨架構建生成新 `.md`，第四次敘述寫作填入散文。隨後把 LOREBOOK 編輯筆記中的“Vault 未登記”改為 `[[術語]]`，並從 Librarying frontmatter 直接更新 frontmatter。

反饋週期越快、自動化程度越高，運營就越輕，因此建議每季度定期執行。

## 版本管理——git 標籤這一棵單樹

LOREBOOK 運營的第二部分是版本管理。第一次週期以 v1.0 出版後，更新會進入 v1.1、v1.2、v2.0。

版本規則很簡單。v0.x 是出版前的內部版本：v0.1 初稿、v0.2 第一次改寫、v0.3 透過一致性。v1.0 是出版版，v1.x 是出版後的細微更新，v2.0 是新版本。`v_確定`表示一個詞條或資料不再變動，用於插畫委託等後續工作依賴它的時點；`v_最終`則指 `v_確定` 中連最後整合構建審校也透過的狀態。

建議採用基於 git 標籤的單一樹，而不要採用 `v1.0/`、`v1.1/`、`v2.0/` 這樣的目錄分支。目錄分支會讓詞條 `.md` 隨版本重複，違背第 1 章的單一 `.md` 源原則，也會使交叉引用連結在各目錄間斷裂。

git 標籤方式把所有資料放在單一樹中，以 git 的時間點快照決定版本。

```
[專案 LOREBOOK]/  （單一樹，git 管理）
   │
   ├─ git tag v0.1   ← 初稿（Phase 3結束）
   ├─ git tag v0.2   ← 第一次改寫（Phase 4結束）
   ├─ git tag v1.0   ← 出版版（Phase 6印刷完成）
   ├─ git tag v1.1   ← 再版時細調
   └─ git tag v2.0   ← 新版本
```

該方式保證單一 `.md` 原則與整潔的交叉引用。只有當 v2.0 與 v1.x 的部構成發生根本變化——如從十部擴充套件為十四部、重大合併或拆分——時，`v1.x_archive` 與 `v2.0` 的目錄分支才是自然的。其餘情形都用 git 標籤處理。

版本更新時，在 `_進度狀態.md` 中記錄更新內容，例如“v1.1：追加短篇 047、052、061 引用；更新第 9 部 §3 男巫詞條”。這一行記錄會成為跟隨 LOREBOOK 運營時間的資料。讀者可在一處檢視 v1.0 與 v2.0 的變遷；新版本出版時，也可據此分析 v1.x 哪些部分更新最頻繁，並用來決定新版本的部構成。

## 插畫委託的一項原則

運營的第三部分是插畫。第 11 章的 Phase 3、6 已討論委託程式，但運營階段仍會繼續追加插畫。

核心只有一項原則：**插畫應在正文達到 v_確定 後再委託。**

正文尚未確定就委託插畫時，正文更新會迫使插畫再次委託，恢復成本極高。一張插畫所需的人類畫師時間超過一條正文詞條的五倍；如果已透過的草圖之後因正文更新而改變角色外觀，草圖本身也可能被廢棄。為避免這一成本，應在正文越過 v_確定 合格線後才開始委託。

以十二部標準 LOREBOOK、約 450 頁 B5 精裝為例，建議數量為：封面正、背、書脊 3 張；襯頁插畫 2 至 4 張；卷首整頁 1 張；部匯入插畫 12 張；角色插畫 30 至 50 張；全綵章節頁首 12 至 15 張；地圖 5 至 10 張；圖解與圖表 20 至 40 張；正文間小型氛圍插圖 50 至 100 張。可依 IP 調整。

每張插畫都應編寫規格卡，置於該詞條的編輯筆記中。

```html
<!--
插畫規格：
- 型別：部匯入 / 角色 / 地圖 / 圖解
- 篇幅：1p整頁 / 0.5p / 邊欄 / 小插圖
- 描述：[核心視覺；正文中要畫的場景]
- 色調：賽博朋克 / 東方 / 維度彼方 / 日常
- 出場人物：[角色卡 [[姓名]]]
- 色彩：五色調色盤中應強調的顏色
- 委託狀態：未委託 / 委託 / 草圖 / 確認 / 完成
- 畫師：___
-->
```

規格卡是委託資料。建議由 AI 工作夥伴自動起草，再由人審校。每次委託給畫師附上五類資料：出現該視覺的詞條正文 `.md`；角色插畫時的角色卡；包含第 7 章七種音色與五色調色盤的色調指南；三至五張同類色調的外部作品參考；以及必須納入元素的細節清單。

草圖審校分四步：一次草圖確認描述和構圖；二次線稿確認細節和人物；三次上色確認色調與色彩；最終完稿確認可印刷規格。每一階段由正文作者審校。若插畫與正文衝突，插畫服從正文，因為正文是 SSOT。

## AI 插畫的領域

隨著 AI 影象生成進入市場，LOREBOOK 插畫需要一項政策：哪些可以委託，哪些必須留給人類畫師。

**推薦**用於畫師委託前的概念草圖、情緒板與參考圖、地圖和圖解圖表——後者因準確性重要，須經人類審校。**有條件允許**用於正文間小氛圍插圖，僅限系列色調與角色一致性已確立時；也可用於部匯入和章節頁首，但必須經人類畫師後處理。**不推薦或禁止**用於角色插畫與封面。主要人物的外觀一致性難以保持；封面正、背、書脊則在營銷、版權、識別性上都是人類畫師專屬領域。

有六項運營規則：AI 生成插畫須在版權頁標明模型、提示詞與後處理者；與角色卡或正文描述衝突時廢棄；收入本卷的插畫全部經人類插畫師後處理；同一角色或地點反覆出現時接受人類一致性審校；禁止使用訓練資料來源不明的模型；概念、氛圍、圖解可允許 AI 輔助，但封面和角色保留為人類專屬。

這些規則劃分了畫師的領域與 AI 的領域。各 IP 可以調整，但封面與主要角色由人類專屬，幾乎是所有 IP 都應維持的合格線。

## 更新週期——季度與五年

LOREBOOK 運營的更新週期分為兩層：**季度更新**與**新版本出版**。

季度更新每季度一次，約兩三天完成。自動檢查用一至兩小時檢查失效連結和 frontmatter 一致性；Vault 同步用半天至一天進行新詞反饋和更名批次替換；角色卡同步用半天更新人物卡。短篇一致性驗證用半天至一天檢查該季度出版的短篇資料並增加短篇引用框。附錄自動再生成用一至兩小時重建術語表和索引；最後用一至兩小時在 `_進度狀態.md` 記錄變更並標記 git 標籤。這個季度工作維持 LOREBOOK 的運營。

新版本出版是運營五年或十年後以 v2.0 進行的大週期。整理 v1.x 運營資料需一至兩個月，收集季度更新並分析模式；重新審視部構成需一個月，重新應用第 4 章程式決定 v2.0 的形態；v2.0 工作本身用三至六個月重組目錄和追加新資料；v2.0 的 Phase 4 至 6 再用三至五個月進行一致性驗證、附錄編寫和網頁/出版打包。最後以新封面和新 ISBN 出版 v2.0，v1.x 保留為 archived。

這一新版本週期約八至十四個月，與第一次週期相近，但因從 v1.x 資料開始而較輕。

## 五年後的樣子

簡要看看虛構 IP“○○”運營五年後的樣子。LOREBOOK 本卷從 2026 年 v1.0 開始：B5 精裝 450 頁，首印 1,000 冊；2026 年 v1.1 細調並再印 500 冊；2027 年 v1.2 追加短篇引用並再印 500 冊；2028 年 v2.0 從十二部擴充套件到十四部，出版新書 1,500 冊。

別卷短篇集從 2026 年三十篇、600 頁，成長為 2027 年追加五十篇、800 頁，再到 2028 年一百篇、1,000 頁。網頁公開版可在 `lorebook.example.com/○○` 訪問，每季度更新，五年累計頁面瀏覽約五十萬。Vault 從第一次週期的 1,200 個詞條增加到五年後的 1,800 個，Obsidian 圖譜中的節點連線也更加密集。

姐妹工具的產出也一同成長：角色卡五年累計到一百人，短篇積累到一百篇，多語言出版增加一本英文版（2027）和一本日文版（2028）。

這一五年圖景自然顯示，LOREBOOK 不是做一次就結束的資料。每季度兩三天的小更新與每五至十年一次的新版本週期，就是 LOREBOOK 的運營形態。書架上的一本書不再是靜止的資料，而是每季度更新、在五年後經歷下一次大型轉換、與 IP 時間一同前行的活資料。
