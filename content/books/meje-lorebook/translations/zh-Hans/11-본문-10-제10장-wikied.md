# 第10章　Phase 0～3——从协议到试点部的前半程

设想一场电影的第一次编剧室会议。编剧们围坐的桌上放着一份打印PDF，封面写着 *NDA* 与 *Show Bible v1.0*。会议的第一个小时用来共同确认各页包含什么；此后，整季的正式工作才能开始。LOREBOOK 项目的第一个月与此相似。它不是撰写正文的时间，而是把一个 IP 的所有决定汇集到同一份文档中的时间。

Phase 0 的产物不是一本书，而是一组协议。

最初2至4周不写正文，而是把如何处理1,500个词条的决定汇集成约10至20页的文档。它会成为随后9至14个月工作的 SSOT；只有先整理好它，骨架、试点部与扩展才能依次展开。

*Phase* 一词借自游戏开发现场，常用于前一阶段产物成为后一阶段输入、依赖关系清楚的工作。LOREBOOK 分为Phase 0至6七个阶段，每一阶段的验收线就是进入下一阶段的门槛。本章讨论前四个：协议（0）、骨架（1）、试点部（2）与扩展（3），即最初约4至6个月。

## 七个Phase的时间流

Phase 0 用2至4周制作协议。Phase 1 是1至2周的骨架构建，自动生成1,500个空的 LOREBOOK 文件并转换 frontmatter。Phase 2 用3至4周制作一个试点部并由主审验收。最长的Phase 3用4至6个月撰写其余11部。Phase 4用1个月验证一致性，Phase 5用1个月制作附录，Phase 6用2个月转换出版版与网页版。

总计9至14个月。纯手工作业需要18至24个月；与 AI 工作伙伴协作，可缩短到接近一半。

每个词条用六种标记追踪：☐未写，◐仅元数据，◑散文初稿，◕散文完成，●含交叉引用的完成稿，✓校对与验收完成。一个`_进度状态.md`文件每周更新1,500个词条的状态，Phase门槛也据此判断。

本章讲0、1、2、3；下一章讲4、5、6。

## Phase 0——用一个月撰写协议

Phase 0需要2至4周，是因为所有决定并非权重相同，彼此又有依赖关系；前一项未完成，就不能进入下一项。

首先要做的是**输入确认**。检查Vault的1,500个词条、13列frontmatter、九分类与世界观轴排列，以及Obsidian图谱视图是否正常。输入不足时，就在Phase 0内返回Librarying补强。13列有缺项，回到第二次整合；九分类或世界观轴未定，回到第一次循环。输入整理完毕，才能确定部结构与语调指南。

权重最大的两项决定是**部结构**与**语调指南**。部结构运用第4章程序，从10部、12部、14部型中选择，并依次检查IP规模、世界观轴分布、类型行动者、维度多层性与短篇资产。语调指南把第7章的七种声音映射到各部，并按IP性质调整比例，另存为约2至3页的Markdown供所有部参考。两者决定书的**身份**，也是Phase 2试点部的验收依据。

下一组是三项政策。**SSOT层级**规定资料冲突时的优先顺序，标准为角色表、主审确认的短篇、LOREBOOK、Vault散文、规则书；某些IP会把规则书上移，外语原作也可能居于最高位。**短篇分离政策**按第9章在另册出版（推荐）、并入第11部、引用框三种方案中选择。**LOREBOOK类型**从第2章五种变体A至E中选择。多数IP采用标准百科型A，若采用其他类型，则在协议中明确差异。这三项决定书的**政策**。

最后是两项实务决定。**出版规格**先确定标准B5开本、约450页、插图预算、装订、封面加工与豪华版选项的大方向，再在出版阶段精调。**日程与预算协议**决定9至14个月内各Phase如何分配时间。即使不同IP有所调整，也必须把整体安排写入协议，以便项目中途检查。

八项决定合成约10至20页的Markdown文档，称为**Bible**。Phase 1至6的一切工作均以它为依据，出现问题也总要返回Bible。若试点部审查发现语调偏差，就修改语调指南；若Phase 3暴露部结构不合理，就重新作出结构决定。

这里还要引入**主审**。每个IP设一人，作为本LOREBOOK的最终决策者，负责部结构、验收、出版规格与每一Phase的GO/NO-GO门槛。个人项目由制作者兼任，位置近似电视剧的总制作人或游戏工作室总监。

## Phase 1——构建空骨架的时间

Phase 1几乎完全自动化。1至2周内构建空的LOREBOOK骨架，人主要负责验收。

七步流程大多是机械操作。先创建作为Obsidian Vault的根目录`[项目 LOREBOOK]/`，在`_Spec.md`中整理Bible的关键决定。再创建从`00 序言`到`12 附录`的12个部目录及各自空的`README.md`，随后创建每部的章目录与空的章README。

核心自动化是创建1,500个空词条Markdown。每个文件的frontmatter由Librarying的13列自动转换，正文保持空白。把13列CSV交给AI工作伙伴，约5分钟即可生成1,500个文件。最后创建根README，并用1,500个☐初始化`_进度状态.md`。

人工部分是检查frontmatter转换：约30分钟随机打开50个样本，确认结果准确。

产物是空的LOREBOOK骨架。Obsidian侧栏排列着1,500个Markdown文件，图谱视图显示由wikilink连接的节点，但节点内部尚无正文。Phase 2至3将逐一填入内容。

## Phase 2——一部决定整本书的命运

Phase 2是项目最大的分岔点。完成一个试点部并由主审验收。GO则进入Phase 3；NO-GO则更新Bible后重写试点。

若带着缺陷进入Phase 3，同一缺陷会扩散到12部，直到Phase 4一致性验证时才暴露；此时所有资料已经写完，恢复成本极高。反之，试点部若准确跨过验收线，其语调、SSOT引用和对绝对禁项的规避会自然延伸到其余各部。一部的决定定义全书格式。

试点通常选第2部“世界法则”或第4部“地理”，以最能显现IP身份者为准。虚构IP“○○”适合第4部，因为巫俗、料理与日常的混合最清晰地体现在城市、街区和生活景观中。若IP由SF或奇幻的魔法机制定义，则第2部更合适。D&D《Forgotten Realms》这类大型IP以魔法学派作为新设定的先导，也是因为机制决定IP性质。

试点流程有六步：选择试点部；撰写部README与全部章README；先把连接最多的3至5个hub词条写成散文；按第7至9章验证语调、结构和SSOT引用模式并反馈到语调指南；进行部级一致性检查，查找死链与五项绝对禁项；最后由主审作出Phase 3的GO/NO-GO决定。

六步约需3至4周，其中hub散文化与主审验收耗时最多。

是否合格看五项：部README能否作为指南；全部章README是否完成；核心3至5个词条是否达到◕以上；语调、结构、SSOT引用是否符合第7至9章；五项绝对禁项是否为0。五项全✓即GO，任何一项未达即NO-GO。

NO-GO时分析原因来自Bible还是执行。若来自Bible，更新后重写；若来自执行，则局部或整体重写。一部的决定左右整个项目，不能草率处理。

也可以不做试点而同时推进12部，但不推荐，因为相同缺陷会铺满全书并延续到Phase 4，难以恢复。

电影制作中的**试播集**承担同样功能。它在整季开始前确定语调、人物与导演方式。《Lost》试播集成本约1,400万美元，接近普通单集的五倍；《Game of Thrones》试播集曾整集重拍，原因都在于一集会决定之后6至8季。LOREBOOK试点部也先检验IP的语调、结构与一致性，再让其余11部沿同一路径推进。

## Phase 3——4至6个月的正式写作

Phase 3是最长阶段，撰写试点之外的11部。每部约2至3周，总计4至6个月。

顺序不是随机的，而是按书的身份逐渐稳定来安排。首先写第1、2、3部（世界语调、世界法则、历史与年表），先确定身份。随后写第5、6、4部（社会、日常、地理），填充体量最大且稳定的生活骨架。再写第7、9、10部（行动者、图鉴），在人物与势力SSOT稳定后整理为卡片；卡片较轻，即使前文修改也容易恢复。最后写第8部（维度与空间图鉴），因为它最具挑战，必须吸收其他部的一致性。

顺序即1 → 2 → 3 → 5 → 6 → 4 → 7 → 9 → 10 → 8。不同IP可稍作调整，但大方向是**身份 → 日常骨架 → 卡片化 → 整合**。

每部重复试点流程：撰写部README与章README；从hub词条开始散文化；撰写标准词条与卡片词条；进行部级一致性检查；按实际篇幅与状态更新部README。五步共重复11次。

插图委托同步进行。一部达到◕后才开始委托。正文未定就先委托，正文更新时必须重新下单，恢复成本很大，因此标准原则是**正文v_确定后委托**。

建议每周更新`_进度状态.md`。某部延迟时能立刻发现，以每周的小检查防止4至6个月的长项目发生大偏移。

典型的五个月安排如下：第1周末，第1部5个hub达到◑；第2周增加5个标准词条；第3周卡片达到◕，第1部过线。第4周开始第2部，同时外包第1部7至10幅插图。若第2部发现语调不符，当场决定重写。第8周第2部过线，第12周第3部，第16周第5部，第20周第6部，第23周加入第4部的综合概览词条。每周固定约一小时检查，可避免重大事故。

产物是所有部达到●至✓的LOREBOOK正文。1,500个词条全部有散文与wikilink，插图委托完成80%以上，Obsidian图谱显示1,500个节点与5,000至10,000条链接。但此时仍未最终完成；Phase 4至6还要依次处理一致性验证、附录与网页/出版封装。

## 每个Phase的进入门槛

明确每一Phase的验收线，构成项目的秩序。

从Phase 0进入1，须完成输入确认与Bible，并由主审检查。从1进入2，自动结构检查验证目录骨架、Spec、总README与全项目为☐或◐的`_进度状态.md`。从2进入3的门槛最严格：试点部达到◕至●，语调指南已更新，主审下达GO。从3进入4，则要求所有部全部词条达到◕以上且插图委托已启动，自动检查与主审验收共同进行。

未通过就返回当前Phase。不建议临时绕行，因为不足会扩散到下一阶段，令恢复成本剧增。因此门槛必须执行。

## 虚构IP“○○”的Phase 0～3案例

Phase 0约3周。输入确认发现Vault已有1,200个词条。因有60名角色，采用把第9、10部分开的12部型；七种声音按部映射，并提高巫俗领域的图鉴体与现场体比重。SSOT顺序为角色表 → 短篇 → LOREBOOK → Vault → 规则书。30篇短篇另册出版，类型采用标准百科型A。出版规格为B5精装约450页、插图预算1,500万韩元，Phase 0至6总计11个月。产物是一份18页Bible。

Phase 1约1.5周。创建`[○○_LOREBOOK]/`、`_Spec.md`、12个部目录、每部3至6章、1,200个已自动转换frontmatter的空词条、总README，以及全词条为◐的`_进度状态.md`。产物为空骨架。

Phase 2约4周。选择第4部“地理”作试点，因为巫俗、料理与日常混合的身份在城市和街区最鲜明。完成第4部README与5个章README后，把四个hub（六风格地区概览、梅里屯、海原餐饮街、旧巫堂街）写成散文。语调、结构与SSOT引用检查确认死链0、绝对禁项0。主审指出第四个词条部分语调偏差，局部重写后获得GO。产物是第4部达到◕至●。

Phase 3约5个月，顺序为第1部（3周）→ 2（4周）→ 3（4周）→ 5（3周）→ 6（3周）→ 7（3周）→ 9（4周）→ 10（3周）→ 8（3周）。每部遵循README → hub → 标准词条 → 卡片词条 → 一致性检查 → README更新。第1部达到◕时开始委托插图，五个月内完成90%；每周五更新`_进度状态.md`。产物是所有部达到●至✓。

至此LOREBOOK正文全部写完。下一步是验证一致性、制作附录，并转换为出版版与公开网页版。
