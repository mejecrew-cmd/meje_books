# 第13章　一致性、版本与插画——活着的 LOREBOOK

从 1987 年第一版到 2024 年第五版，D&D《被遗忘的国度》在约三十七年间经历了五次重大修订。期间约有两百本短篇集、小说和战役模组出版，每一次都有新人物、新城市、新事件积累到 LOREBOOK 中。费伦从 1372 DR 走到 1492 DR；原本是一座城市的扎卡拉变成新兴势力的根据地；法师伊尔明斯特的一生由新作者重新讲述。这三十七年的流动正是 LOREBOOK **活着**的范本。一个 IP 的资料不会在出版后静止，而会延续为每季度、每年的小更新，以及每五年、十年的大型新版本。

v1.0 出版后，每季度约两三天的定期更新使 LOREBOOK 保持活力。

IP 运营期间会有新短篇、角色更新、Vault 增量更新、多语言版追加、词条更名。每次变化，LOREBOOK 都随之跟进。每季度一次的小更新与每五至十年一次的大型新版本构成运营的两层。若第一次周期把书的**材料**变成了**书**，运营阶段则让这本书成为**跟随时间的资料**。

本章讨论这种运营：一致性验证、版本管理、插画委托、AI 插画政策、更新周期，以及五年运营后的面貌。它从第一次周期结束之处开始，一直看到运营扎根的阶段。

## 一致性验证——每季度的小检查

运营的第一部分是一致性验证。第 11 章将其作为 Phase 4 处理；在运营阶段，它会在每季度、新短篇出版后、角色卡更新后定期执行。

它由三个阶段组成。第一阶段是**自动检查**。审校脚本在首次检查中找出失效链接、重复词条、frontmatter 一致性、绝对禁止项和名词 grep。约需一至两小时；每季度一次、新短篇出版后一周内、新赛季或大型 IP 更新后一月内、角色卡 v_最终更新后一月内执行。

第二阶段是**人工抽样审校**。自动检查通过后，由人直接查看 1,500 个词条中的三十至五十个随机样本，修正自动检查无法发现的细微问题。它约需一至两天，并与自动检查同时进行。样本应按部均匀分配——每部约三至五个——遵循 hub、标准、卡片型 1:2:7 的比例，并优先审校最近更新的词条。

最后是**集成构建审校**。这一阶段负担较大，不在每季度执行，只在第一次周期出版前、v2.0 等新版本出版前，以及季节转换、世界观重大变动等大型 IP 更新后进行。由人构建整个 LOREBOOK，将其作为一本书通读，并对是否达到合格线作出最终判断。

三个阶段结合起来，形成防止运营中资料偏移的多层检查体系：自动检查每季度先行抓取问题，人工样本补足细微部分，集成构建在大转折点检视全书。

## 与 Vault 同步

LOREBOOK 运营的核心是与 Vault 同步。Librarying 阶段的产出是 LOREBOOK 的输入，因此 Vault 更新时，LOREBOOK 也应同步更新。它分为两种形式。

第一种是**从 Vault 到 LOREBOOK 的批量更新**。Vault 中某个词条改名或新增词条时，LOREBOOK 接收该变化并更新。

以虚构 IP“○○”为例。假设 Librarying 第二次整合把一个词条从“黑孔”改为“暗渊”。Librarying 第三次重建生成新的 `.md`，旧 `.md` 以 `version=废止` 隔离。将 LOREBOOK 正文和编辑笔记中的旧术语批量替换为新术语，可使用 `sed` 自动化或 Obsidian 批量修改。将旧术语加入 LOREBOOK frontmatter 的 `aliases`，读者以旧写法搜索时仍能找到新词条。最后在 `_检查清单.md` 记录：正文旧术语残留 0 项、aliases 登记 100%、附录一术语表已记录变更历史。整个流程约一至两小时完成。

第二种是**新术语的双向反馈**。LOREBOOK 工作中可能出现 Vault 尚未收录的新术语，例如新短篇自然引入的新词。可在 LOREBOOK 词条编辑笔记中标记“Vault 未登记：[术语名]”；在下一季度 Librarying 的第一、第二次增量周期中，将该术语登记到九分类之一并填写十三字段。第三次骨架构建生成新 `.md`，第四次叙述写作填入散文。随后把 LOREBOOK 编辑笔记中的“Vault 未登记”改为 `[[术语]]`，并从 Librarying frontmatter 直接更新 frontmatter。

反馈周期越快、自动化程度越高，运营就越轻，因此建议每季度定期执行。

## 版本管理——git 标签这一棵单树

LOREBOOK 运营的第二部分是版本管理。第一次周期以 v1.0 出版后，更新会进入 v1.1、v1.2、v2.0。

版本规则很简单。v0.x 是出版前的内部版本：v0.1 初稿、v0.2 第一次改写、v0.3 通过一致性。v1.0 是出版版，v1.x 是出版后的细微更新，v2.0 是新版本。`v_确定`表示一个词条或资料不再变动，用于插画委托等后续工作依赖它的时点；`v_最终`则指 `v_确定` 中连最后集成构建审校也通过的状态。

建议采用基于 git 标签的单一树，而不要采用 `v1.0/`、`v1.1/`、`v2.0/` 这样的目录分支。目录分支会让词条 `.md` 随版本重复，违背第 1 章的单一 `.md` 源原则，也会使交叉引用链接在各目录间断裂。

git 标签方式把所有资料放在单一树中，以 git 的时间点快照决定版本。

```
[项目 LOREBOOK]/  （单一树，git 管理）
   │
   ├─ git tag v0.1   ← 初稿（Phase 3结束）
   ├─ git tag v0.2   ← 第一次改写（Phase 4结束）
   ├─ git tag v1.0   ← 出版版（Phase 6印刷完成）
   ├─ git tag v1.1   ← 再版时细调
   └─ git tag v2.0   ← 新版本
```

该方式保证单一 `.md` 原则与整洁的交叉引用。只有当 v2.0 与 v1.x 的部构成发生根本变化——如从十部扩展为十四部、重大合并或拆分——时，`v1.x_archive` 与 `v2.0` 的目录分支才是自然的。其余情形都用 git 标签处理。

版本更新时，在 `_进度状态.md` 中记录更新内容，例如“v1.1：追加短篇 047、052、061 引用；更新第 9 部 §3 男巫词条”。这一行记录会成为跟随 LOREBOOK 运营时间的资料。读者可在一处查看 v1.0 与 v2.0 的变迁；新版本出版时，也可据此分析 v1.x 哪些部分更新最频繁，并用来决定新版本的部构成。

## 插画委托的一项原则

运营的第三部分是插画。第 11 章的 Phase 3、6 已讨论委托程序，但运营阶段仍会继续追加插画。

核心只有一项原则：**插画应在正文达到 v_确定 后再委托。**

正文尚未确定就委托插画时，正文更新会迫使插画再次委托，恢复成本极高。一张插画所需的人类画师时间超过一条正文词条的五倍；如果已通过的草图之后因正文更新而改变角色外观，草图本身也可能被废弃。为避免这一成本，应在正文越过 v_确定 合格线后才开始委托。

以十二部标准 LOREBOOK、约 450 页 B5 精装为例，建议数量为：封面正、背、书脊 3 张；衬页插画 2 至 4 张；卷首整页 1 张；部导入插画 12 张；角色插画 30 至 50 张；全彩章节页眉 12 至 15 张；地图 5 至 10 张；图解与图表 20 至 40 张；正文间小型氛围插图 50 至 100 张。可依 IP 调整。

每张插画都应编写规格卡，置于该词条的编辑笔记中。

```html
<!--
插画规格：
- 类型：部导入 / 角色 / 地图 / 图解
- 篇幅：1p整页 / 0.5p / 边栏 / 小插图
- 描述：[核心视觉；正文中要画的场景]
- 色调：赛博朋克 / 东方 / 维度彼方 / 日常
- 出场人物：[角色卡 [[姓名]]]
- 色彩：五色调色板中应强调的颜色
- 委托状态：未委托 / 委托 / 草图 / 确认 / 完成
- 画师：___
-->
```

规格卡是委托资料。建议由 AI 工作伙伴自动起草，再由人审校。每次委托给画师附上五类资料：出现该视觉的词条正文 `.md`；角色插画时的角色卡；包含第 7 章七种音色与五色调色板的色调指南；三至五张同类色调的外部作品参考；以及必须纳入元素的细节清单。

草图审校分四步：一次草图确认描述和构图；二次线稿确认细节和人物；三次上色确认色调与色彩；最终完稿确认可印刷规格。每一阶段由正文作者审校。若插画与正文冲突，插画服从正文，因为正文是 SSOT。

## AI 插画的领域

随着 AI 图像生成进入市场，LOREBOOK 插画需要一项政策：哪些可以委托，哪些必须留给人类画师。

**推荐**用于画师委托前的概念草图、情绪板与参考图、地图和图解图表——后者因准确性重要，须经人类审校。**有条件允许**用于正文间小氛围插图，仅限系列色调与角色一致性已确立时；也可用于部导入和章节页眉，但必须经人类画师后处理。**不推荐或禁止**用于角色插画与封面。主要人物的外观一致性难以保持；封面正、背、书脊则在营销、版权、识别性上都是人类画师专属领域。

有六项运营规则：AI 生成插画须在版权页标明模型、提示词与后处理者；与角色卡或正文描述冲突时废弃；收入本卷的插画全部经人类插画师后处理；同一角色或地点反复出现时接受人类一致性审校；禁止使用训练数据来源不明的模型；概念、氛围、图解可允许 AI 辅助，但封面和角色保留为人类专属。

这些规则划分了画师的领域与 AI 的领域。各 IP 可以调整，但封面与主要角色由人类专属，几乎是所有 IP 都应维持的合格线。

## 更新周期——季度与五年

LOREBOOK 运营的更新周期分为两层：**季度更新**与**新版本出版**。

季度更新每季度一次，约两三天完成。自动检查用一至两小时检查失效链接和 frontmatter 一致性；Vault 同步用半天至一天进行新词反馈和更名批量替换；角色卡同步用半天更新人物卡。短篇一致性验证用半天至一天检查该季度出版的短篇资料并增加短篇引用框。附录自动再生成用一至两小时重建术语表和索引；最后用一至两小时在 `_进度状态.md` 记录变更并标记 git 标签。这个季度工作维持 LOREBOOK 的运营。

新版本出版是运营五年或十年后以 v2.0 进行的大周期。整理 v1.x 运营资料需一至两个月，收集季度更新并分析模式；重新审视部构成需一个月，重新应用第 4 章程序决定 v2.0 的形态；v2.0 工作本身用三至六个月重组目录和追加新资料；v2.0 的 Phase 4 至 6 再用三至五个月进行一致性验证、附录编写和网页/出版打包。最后以新封面和新 ISBN 出版 v2.0，v1.x 保留为 archived。

这一新版本周期约八至十四个月，与第一次周期相近，但因从 v1.x 资料开始而较轻。

## 五年后的样子

简要看看虚构 IP“○○”运营五年后的样子。LOREBOOK 本卷从 2026 年 v1.0 开始：B5 精装 450 页，首印 1,000 册；2026 年 v1.1 细调并再印 500 册；2027 年 v1.2 追加短篇引用并再印 500 册；2028 年 v2.0 从十二部扩展到十四部，出版新书 1,500 册。

别卷短篇集从 2026 年三十篇、600 页，成长为 2027 年追加五十篇、800 页，再到 2028 年一百篇、1,000 页。网页公开版可在 `lorebook.example.com/○○` 访问，每季度更新，五年累计页面浏览约五十万。Vault 从第一次周期的 1,200 个词条增加到五年后的 1,800 个，Obsidian 图谱中的节点连接也更加密集。

姐妹工具的产出也一同成长：角色卡五年累计到一百人，短篇积累到一百篇，多语言出版增加一本英文版（2027）和一本日文版（2028）。

这一五年图景自然显示，LOREBOOK 不是做一次就结束的资料。每季度两三天的小更新与每五至十年一次的新版本周期，就是 LOREBOOK 的运营形态。书架上的一本书不再是静止的资料，而是每季度更新、在五年后经历下一次大型转换、与 IP 时间一同前行的活资料。
