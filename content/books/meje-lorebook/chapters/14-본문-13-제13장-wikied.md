# 제13장. 정합성·버전·일러스트 - 살아 움직이는 LOREBOOK

D&D 〈Forgotten Realms〉는 1987년 1판부터 2024년 5판까지 약 37년에 걸쳐 다섯 차례 큰 개정을 거쳐 왔습니다. 그 사이에 약 200권의 단편집과 소설, 캠페인 모듈이 출간되었고, 그때마다 새 인물과 새 도시, 새 사건이 LOREBOOK에 쌓였습니다. 페이룬의 1372 DR이 1492 DR로 옮겨 가고, 한 도시였던 자카라가 다른 신흥 세력의 본거지로 바뀌고, 한 마법사 엘민스터의 평생이 새 작가들의 손에서 다시 풀어집니다. LOREBOOK이 **살아 있다**는 말의 본보기가 바로 그 37년의 흐름입니다. 한 IP의 자료가 출간 직후 정지되는 것이 아니라, 매 분기와 매 해의 작은 갱신과 5년·10년 주기의 큰 차기판으로 이어집니다.

v1.0 출간 후 매 분기 약 2~3일의 정기 갱신이 LOREBOOK을 살아 있게 합니다.

본 IP가 운영되는 동안 새 단편이 나오고, 캐릭터가 갱신되고, Vault가 증분 업데이트되고, 다국어판이 추가되고, 한 표제어의 명칭이 바뀝니다. 그때마다 LOREBOOK이 그 변경을 따라갑니다. 매 분기 한 번의 작은 갱신과 5~10년에 한 번의 큰 차기판이 운영의 두 층위를 이룹니다. 1차 주기에서 책의 **재료**가 **책**으로 변환됐다면, 운영 단계에서는 그 책이 **시간을 따라가는 자료**가 됩니다.

이 회는 그 운영을 다룹니다. 정합성 검증, 버전 관리, 일러스트 발주, AI 일러스트 정책, 갱신 주기, 그리고 5년 운영의 모습. 1차 주기가 끝나는 지점에서 시작해 운영이 자리를 잡는 단계까지 짚어 봅니다.

## 정합성 검증 - 매 분기의 작은 점검

운영의 첫 부분이 정합성 검증입니다. 11장에서 Phase 4로 다뤘지만, 운영 단계에서는 매 분기, 새 단편 출간 후, 캐릭터 시트 갱신 후에 정기적으로 실행됩니다.

세 단계로 짜집니다. 첫 단계는 **자동 검사**입니다. 검수 스크립트가 1차 검사로 죽은 링크, 중복 표제어, frontmatter 일치성, 절대 금지, 명사 grep을 잡아 줍니다. 약 1~2시간이 걸리고, 매 분기 한 번, 새 단편 출간 후 1주 이내, 새 시즌·IP 갱신 후 1개월 이내, 캐릭터 시트 v_최종 갱신 후 1개월 이내에 정기적으로 실행됩니다.

둘째는 **수동 샘플 검수**입니다. 자동 검사를 통과한 뒤 사람이 1,500 항목 중 무작위 샘플 30~50개를 직접 살핍니다. 자동 검사가 잡지 못한 미묘한 부분을 사람이 손보는 단계로, 약 1~2일이 걸리며 자동 검사 시점에 함께 진행됩니다. 샘플은 부별로 균등하게 분배하고(한 부에 약 3~5개), hub·표준·카드형 비율(1:2:7)을 따르며, 최근 갱신된 항목을 우선 검수합니다.

마지막은 **통합 빌드 검수**입니다. 부담이 큰 단계라 매 분기마다 진행하지 않고 세 시점에서만 합니다. 1차 주기 출간 직전(Phase 6의 시범 인쇄본 검수), 차기판(v2.0) 출간 직전, 그리고 시즌 전환이나 세계관의 큰 변동 같은 큰 IP 갱신 후입니다. 사람이 LOREBOOK 전체를 통합 빌드해 한 권으로 통독하고, 책이 합격선을 넘었는지에 대한 마지막 판단을 내립니다.

세 단계가 결합해 운영 중에 자료가 어긋나지 않도록 다층으로 점검하는 체계를 이룹니다. 자동 검사가 매 분기 1차로 잡고, 수동 샘플이 미묘한 부분을 보완하고, 통합 빌드가 큰 변곡점에서 책 전체를 점검하는 식입니다.

## Vault와의 동기화

LOREBOOK 운영의 핵심은 Vault와의 동기화입니다. 라이브러링 단계의 산출물이 LOREBOOK의 입력이므로, Vault가 갱신되면 LOREBOOK도 함께 갱신됩니다. 두 형태로 갈립니다.

첫 형태는 **Vault에서 LOREBOOK으로의 일괄 갱신**입니다. Vault에서 한 표제어의 명칭이 변경되거나 새 표제어가 등록되면 LOREBOOK이 그 변경을 받아 갱신합니다.

가공 IP "○○"의 사례를 들어 봅니다. 라이브러링 2차 통합에서 한 표제어 명칭이 "흑공"에서 "암연"으로 바뀐다고 가정합시다. 라이브러링 3차 재빌드로 신규 .md가 생성되고 구 .md는 version=폐기로 격리됩니다. LOREBOOK 본문과 편집 메모에서는 구 용어명을 신규 용어명으로 일괄 치환합니다(sed 자동화 또는 옵시디안 일괄 변경). LOREBOOK frontmatter의 aliases에 구 용어명을 등록해 두면, 독자가 구 표기로 검색해도 신규 표기 항목이 잡힙니다. 마지막으로 _체크리스트.md에 구 용어 본문 잔존 0건, aliases 등록 100%, 부록 1 용어집에 변경 이력 기록을 추가합니다. 이 한 흐름이 약 1~2시간에 끝납니다.

둘째 형태는 **신규 용어의 양방향 환류**입니다. LOREBOOK 작업 중 Vault에 없는 신규 용어가 등장할 수 있습니다. 새 단편의 새 용어가 자연스럽게 들어오는 경우입니다. LOREBOOK 항목 편집 메모에 "Vault 미등록: [용어명]" 태깅을 두고, 다음 분기의 라이브러링 1·2차 증분 주기에 그 용어를 9분류 중 하나로 등록하고 13컬럼을 작성합니다. 라이브러링 3차 골격 빌드로 신규 .md가 생성되고, 4차 서술 집필로 산문이 채워집니다. 그 뒤 LOREBOOK 항목 편집 메모를 갱신해 "Vault 미등록"을 "[[용어]]"로 바꾸고, frontmatter도 라이브러링 frontmatter를 그대로 받아 갱신합니다.

환류 주기가 빠르고 자동화될수록 운영이 가벼워지므로, 매 분기 정기 실행을 권장합니다.

## 버전 관리 - git 태그라는 단일 트리

LOREBOOK 운영의 두 번째 부분은 버전 관리입니다. 1차 주기가 v1.0으로 출간된 후 v1.1·v1.2·v2.0의 갱신이 진행됩니다.

버전 표기 규약은 단순합니다. v0.x는 출간 전 작업 단계의 내부 버전입니다(v0.1 초고 → v0.2 1차 개작 → v0.3 정합 통과). v1.0이 출간본, v1.x가 출간 후 미세 갱신, v2.0이 차기판입니다. v_확정은 한 항목 또는 한 자료가 더 이상 변경되지 않는 상태로, 일러스트 발주처럼 후속 작업이 의존하는 시점에 사용합니다. v_최종은 v_확정 중에서도 마지막 통합 빌드 검수까지 통과한 상태를 가리킵니다.

운영 방식은 git 태그 기반 단일 트리를 권장합니다. 디렉토리 분기 방식(v1.0/, v1.1/, v2.0/ 식의 폴더 분리)은 권장하지 않습니다. 항목 .md 파일이 버전마다 중복되어 단일 .md 소스 원칙(1장)을 어기고, 교차 참조 링크가 디렉토리마다 깨지기 때문입니다.

git 태그 방식은 단일 트리에 모든 자료가 있고 git의 시점 스냅샷이 버전을 결정합니다.

```
[프로젝트 LOREBOOK]/  (단일 트리, git 관리)
   │
   ├─ git tag v0.1   ← 초고 (Phase 3 종료)
   ├─ git tag v0.2   ← 1차 개작 (Phase 4 종료)
   ├─ git tag v1.0   ← 출판본 (Phase 6 인쇄 완료)
   ├─ git tag v1.1   ← 재쇄 시 미세 조정
   └─ git tag v2.0   ← 차기판
```

이 방식은 단일 .md 원칙이 보장되고 교차 참조도 깔끔합니다. 다만 v2.0 차기판이 v1.x와 부 구성이 근본적으로 달라지는 경우, 즉 10부에서 14부로의 확장이나 부 통합·분할 같은 큰 변동이 있을 때는 v1.x_archive와 v2.0의 디렉토리 분기가 자연스럽습니다. 그 외에는 모두 git 태그로 처리합니다.

버전 갱신 시에는 _진척상태.md에 갱신 내역을 기록합니다. "v1.1: 단편 047·052·061 인용 추가, 9부 §3 박수무당 항목 갱신" 같은 한 줄 기록입니다. 이 기록이 LOREBOOK 운영의 시간을 따라가는 자료가 됩니다. 독자가 v1.0과 v2.0 사이의 변천을 한 곳에서 확인할 수 있고, 차기판 출간 시에는 v1.x의 어느 부분이 가장 자주 갱신되었는지를 분석해 차기판 부 구성을 결정하는 자료로 사용할 수 있습니다.

## 일러스트 발주의 원칙 하나

운영의 세 번째 부분은 일러스트입니다. 11장의 Phase 3·6에서 발주 절차를 다뤘지만, 운영 단계에서도 일러스트 추가가 이어집니다.

원칙 하나가 핵심입니다. **일러스트는 본문 v_확정 후 발주**한다는 것입니다.

본문이 아직 확정되지 않은 상태에서 일러스트가 발주되면 본문 갱신 시 일러스트도 다시 발주해야 해서 회복 비용이 큽니다. 일러스트 한 장에 들어가는 인간 작가의 시간이 본문 한 항목의 다섯 배 이상이고, 일러스트 시안이 통과한 뒤 본문이 갱신되어 캐릭터 외형이 바뀌면 통과한 시안 자체가 폐기되는 경우도 일어납니다. 이 회복 비용을 막기 위해 본문이 합격선(v_확정)을 넘은 뒤에야 일러스트 발주를 시작하는 것이 표준입니다.

12부형 표준 LOREBOOK ~450p 기준의 일러스트 종류와 권장 수량은 다음과 같습니다. 표지(앞·뒤·책등) 3장, 면지 일러스트 2~4장(지도·세계 풍경), 권두 전면 일러스트 1장, 부 도입 일러스트 12장(각 부의 첫 페이지), 캐릭터 일러스트 30~50장(주요 인물 + 9부 도감), 풀컬러 챕터 헤더 12~15장, 지도 5~10장(전세계·지역·도시·차원계), 도해·다이어그램 20~40장(관계도·계통도·구조도), 작은 분위기 컷 50~100장(본문 사이 작은 삽화). 독자의 IP에 따라 조정됩니다.

각 일러스트마다 스펙 카드를 작성하고, 독자의 .md 항목 편집 메모에 둡니다.

```html
<!--
일러스트 스펙:
- 종류: 부 도입 / 캐릭터 / 지도 / 도해
- 분량: 1p 전면 / 0.5p / 사이드 / 작은 컷
- 묘사: [핵심 시각. 본문에서 어떤 장면을 그릴지]
- 톤: 사이버펑크 / 동양 / 차원 너머 / 일상
- 등장 인물: [캐릭터 시트 [[이름]]]
- 색조: 5색 팔레트 중 어떤 색을 강조
- 발주 상태: 미발주 / 발주 / 시안 / 컨펌 / 완료
- 작가: ___
-->
```

스펙 카드가 일러스트 발주의 자료입니다. AI 작업 동료가 자동으로 작성한 후 사람이 검수하는 흐름이 권장됩니다.

각 일러스트 발주 시 작가에게 동봉하는 자료는 다섯입니다. 해당 시각이 등장하는 항목 본문 .md, 인물 일러스트일 때의 캐릭터 시트, 7장의 7 음색과 5색 팔레트가 담긴 톤 가이드, 같은 톤의 외부 작품 참조 사례 3~5장, 꼭 들어가야 할 요소를 정리한 디테일 체크리스트입니다. 다섯 자료가 갖추어져 있으면 작가가 첫 시안을 효율적으로 짜낼 수 있습니다.

시안 검수는 네 단계입니다. 1차 시안(러프)에서 묘사와 구도를 확인하고, 2차 시안(펜선)에서 디테일과 인물을 확인하고, 3차 시안(채색)에서 톤과 색조를 확인하고, 최종(마감)에서 인쇄 가능 사양을 확인합니다. 각 단계마다 본문 작성자가 검수하고, 본문과 충돌하면 일러스트가 본문에 맞춥니다(본문이 SSOT입니다).

## AI 일러스트의 영역

최근 AI 이미지 생성이 시장에 정착하면서 LOREBOOK 일러스트에도 AI가 들어왔습니다. 어디까지 위임하고 어디서 인간 작가를 두어야 하는지에 대한 정책이 필요합니다.

권장 정책은 일러스트 종류에 따라 다섯 부류로 갈립니다.

**권장**은 컨셉 시안(작가 발주 전 톤 합의용으로 인간 작가에게 동봉), 무드 보드와 참조 컷(본문 반영용 분위기 자료), 지도와 도해, 다이어그램(정확성이 중요한 자료이므로 인간 검수 후 사용)입니다.

**조건부**는 본문 사이의 작은 분위기 컷(시리즈 톤과 캐릭터 일관성이 확보된 경우에만 허용)과 부 도입·챕터 헤더(인간 작가 후처리 의무)입니다.

**비권장 또는 금지**는 캐릭터 일러스트와 표지입니다. 캐릭터 일러스트는 주요 인물의 외형 정합성과 시리즈 일관성을 유지하기 어려워 비권장이고, 표지 앞·뒤·책등은 마케팅과 저작권, 식별성이 모두 인간 작가의 전속 영역이라 금지입니다.

운용 규칙은 여섯입니다. AI 생성 일러스트는 크레딧 페이지에 모델과 프롬프트, 후처리자를 명시하고, 캐릭터 시트나 본문 묘사와 충돌하면 폐기합니다. 본권 수록 일러스트는 모두 인간 일러스트레이터의 후처리를 거치며, 동일 캐릭터나 장소가 여러 일러스트에 등장할 때는 인간 작가의 일관성 검수를 받습니다. 학습 데이터 출처가 불명한 모델은 사용을 금지하고, 컨셉·무드·도해는 AI 보조를 허용하되 표지와 캐릭터는 인간 전속으로 둡니다.

여섯 규칙이 작가의 영역과 AI의 영역을 갈라 둡니다. 독자의 IP에 따라 조정될 수 있지만, 표지와 주요 캐릭터의 인간 전속은 거의 모든 IP에서 유지되는 합격선입니다.

## 갱신 주기 - 분기와 5년

LOREBOOK 운영을 갱신 주기로 정리하면 두 층위로 갈립니다. **분기별 갱신**과 **차기판 출간**입니다.

분기별 갱신은 매 분기 한 번, 약 2~3일에 끝나는 작업입니다. 자동 검사(1~2시간)에서 죽은 링크와 frontmatter 일치성을 점검하고, Vault 동기화(반나절~1일)에서 새 표제어 환류와 명칭 변경 일괄 치환을 진행하고, 캐릭터 시트 동기화(반나절)에서 인물 카드를 갱신합니다. 단편 정합성 검증(반나절~1일)에서는 분기 동안 출간된 단편의 자료를 확인하고 단편 인용 박스를 추가합니다. 부록 자동 재생성(1~2시간)에서 용어집과 색인을 다시 짜고, 마지막으로 변경 이력 기록(1~2시간)에서 _진척상태.md를 갱신하고 git tag를 표시합니다. 매 분기 한 번 진행되는 이 작업이 LOREBOOK의 운영을 유지해 줍니다.

차기판 출간은 5년 또는 10년 운영 후 v2.0 출간으로 진행되는 큰 주기입니다. v1.x 운영 자료 정리(1~2개월)에서 5~10년치의 분기별 갱신 자료를 한 곳에 모으고 갱신 패턴을 분석합니다. 부 구성 재검토(1개월)에서는 4장의 절차를 재적용해 v2.0이 어떤 형이 될지를 결정합니다. v2.0 작업(3~6개월)에서 디렉토리 재구성과 새 자료 추가가 이루어지고, v2.0 Phase 4~6(3~5개월)에서 정합성 검증과 부록 작성, 웹/출판 묶음이 진행됩니다. 마지막으로 v2.0이 출간되는데, 새 표지와 새 ISBN을 달고 v1.x는 archived 상태로 보존됩니다.

차기판 주기는 약 8~14개월에 끝납니다. 1차 주기와 비슷한 시간이지만 v1.x의 자료를 받아 시작하므로 작업이 한층 가볍습니다.

## 5년 후의 모습

가공 IP "○○"이 5년 운영된 후 LOREBOOK이 어떤 모습이 되어 있는지를 짧게 짚어 봅니다.

LOREBOOK 본권은 v1.0 출간(2026)으로 B5 양장 450p 초판 1,000부에서 시작해, v1.1(2026)에서 미세 조정과 재쇄 500부, v1.2(2027)에서 단편 추가 인용과 재쇄 500부, v2.0(2028)에서 12부형에서 14부형 확장과 신간 1,500부로 흘러갑니다.

별권 단편집은 v1.0 단편집(2026)의 단편 30편 600p에서 v1.1 단편집(2027)의 단편 50편 추가 800p로, v2.0 단편집(2028)의 단편 100편 1,000p로 자라갑니다.

웹 공개본은 도메인 lorebook.example.com/○○에서 접근 가능하고 매 분기 갱신되며, 5년 누적 페이지뷰 약 50만에 도달합니다. Vault는 1,200 표제어(1차)에서 1,800 표제어(5년 후)로 증가하고 매 분기 환류 주기를 따라갑니다. 옵시디안 그래프 뷰의 노드 간 연결도 한층 촘촘해집니다.

자매 도구의 산출물도 함께 자라갑니다. 캐릭터 시트는 5년 누적 100명에 이르고, 단편은 100편이 쌓이며, 다국어 발간으로 영문판 1권(2027)과 일본어판 1권(2028)이 더해집니다.

LOREBOOK이 한 번 만들고 끝나지 않는 자료라는 점이 이 5년의 모습에서 자연스럽게 드러납니다. 독자의 IP가 5년 운영에 도달했을 때 이 글의 절차가 어떻게 작동할지를 미리 떠올려 두면 1차 주기의 비중이 한층 실감 있게 다가옵니다.

분기별 2~3일의 작은 갱신과 5~10년에 한 번의 차기판 주기가 LOREBOOK의 운영 형태입니다. 책장에 놓인 한 권이 정지된 자료가 아니라 매 분기 갱신되는 살아 있는 자료가 되고, 5년 뒤 차기판으로 한 번 더 큰 변환을 거치면서 본 IP의 시간을 함께 따라갑니다.
