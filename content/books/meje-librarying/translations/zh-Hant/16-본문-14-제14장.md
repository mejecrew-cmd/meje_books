# 第14章　面對空的 Vault：本書想傳達的事

回到第1章的第一句。無法用一個詞解釋《Star Wars》。原力、絕地、光劍、達斯・維達、反抗軍與銀河帝國不是各自孤立的點，而是編成彼此相連的關係網；那個整體才是《Star Wars》。一個 IP，就是構成它的所有詞彙所形成的關係網。

本書以十四章引導了有意識地建立這個關係網的工作。從四階段管線、九種分類與世界觀軸，到冪等規則、驗證與營運循環，都是把散落在 IP 腦海中的關係網，取出為可掌握的 Obsidian Vault 資料的程序。

現在讀者桌上擺著十三欄 CSV 格式、九分類指南與仍然空白的 Markdown 檔案。檔案之間尚無連線，但讀者已知道：填下第一列的瞬間，第一階段抽取便開始；那一列增加到數百列、在第二階段整合成詞條，第三階段變成 Markdown 檔，第四階段補上敘述，點與點之間便會織起連線，一個 IP 的關係網地圖將在螢幕上展開。

## 既有創作慣例與 Librarying 的差異

過去並非沒有營運 IP 的方法。作家個人的設定筆記、團隊共用文件、Google 試算表、wiki 網站，都以各自方式管理 IP 資訊。

但這些方法共有四個限制：格式不結構化、依賴單一作者、缺乏版本管理、難以移交。設定筆記由作家隨意書寫，人讀得懂，機器卻難以讀取和處理，搜尋、一致性檢查與在其他工具再利用都很困難。真正設定常在作家腦中，文件只保存部分內容，作家不在時許多資訊便不確定。IP 發展中設定雖會改變，卻難追蹤哪個才是最新版本；第三版已修改的內容，第二版設定仍可能留在別處。移交給其他團隊或長時間後重新處理時，許多設定只存在原作者記憶中，因而難以傳承。

Librarying 是對這些限制的方法論回應。十三欄 CSV 與 Markdown 檔案結構預先確定，機器可讀、自動化工具可處理、其他工具可再利用。Vault 不在一人腦中而在檔案系統，第一次接觸 IP 的人打開 Vault 也可確認官方設定。frontmatter 的版本狀態欄與出處清單欄保存每個詞條的版本歷史，冪等規則則在更新時保護既有工作。移交一個 Vault 資料夾，也就同時移交 IP 的官方詞彙與世界觀結構。

這正是 Librarying 改進既有慣例之處。它不是成為更好作家的方法，而是建立能更好管理 IP 的系統的方法。

## 本書想傳達的事

若把十四章縮成一句話：Librarying 不是用 Obsidian 做 wiki，而是製作人建立一份能營運整個 IP 的唯一原本。

差異不在容器，而在切分的方法。即使使用同一個 Obsidian，因以九分類與世界觀軸兩套座標切分，這個 Vault 便不是供搜尋的字典，而是供營運的原本。新短篇、新角色、外語翻譯與百科全書，全都通過這一冊向外產生。

因此 Vault 並非完成時就結束的資料。各項工作產生的新詞彙會藉回流回來，冪等規則保護已寫敘述，同時在其上疊加下一輪。只要回流不停止，IP 就仍活著。

## 製作人的角色：做決定的手

本書始終強調一件事：製作人不是作家。

作家是寫作的人；製作人則是一邊寫、一邊管理、委派、檢查與重寫的人。寫作只是他眾多工作之一。Librarying 是製作人的工作：設計四階段管線，決定要把什麼、如何委派給 AI 工作夥伴，檢查成果並營運 Vault。其中包括直接寫作階段（直接撰寫 hub）、委派階段（leaf 敘述）、確認階段（驗證）。

第2章說過：「做決定的手是人，紡出內容的手是 AI。」讀完十四章，讀者應能具體感到這句話如何運作。九分類要放入哪一類由人決定，世界觀軸由人設計，直接寫 hub、立刻修正 P0 議題、透過回流讓 Vault 持續成長，也都是人的責任。AI 產出而經人檢查通過的內容填滿 Vault；AI 出錯而由人修正的部分形成 Vault 的品質。這種分工良好運作時，一個人難以獨自承擔的規模，便能在現實時間內完成。

## 寫給開始第一輪的讀者

請從小型 IP 開始。第一輪不必以 1,500 個詞條為目標；只選 5 至 10 份核心文件，先做出 300 至 500 個詞條的 Vault。目標是把四個階段完整走過一次；與其從一開始就想完美處理大型 IP 而中途停下，不如把小 IP 做到最後，價值高得多。

最初練習時，世界觀軸可輕量地從三至四個開始。然而進入營運用 Vault 就不同。營運工作流程應在 profiling 階段確定五至七個核心軸，另加「一般」與「未定」；第二階段整合與驗證都在這些允許值內運作。這不是要一開始就猜中完美軸名，而是至少要在工作前寫下準備測試哪些軸。

練習用第一個 Vault 的第二階段整合不必完美。若檔案目的只是熟悉整體流程，即使完成度為 85 至 90%，也可以推進至第三與第四階段；但要交給其他工作的營運用 Vault，合格線不同。同義詞組、代表關鍵字、世界觀軸、版本狀態需達約 95%，且必須通過自動驗證與 handoff 檢查。等待完美以致 Vault 永遠無法完成是問題；把練習用的寬鬆誤認為營運標準也是問題。

第四階段請先直接寫幾個 hub。小型練習 Vault 三至五個即可建立調性基準，營運用首輪則建議五至十個。那些敘述將成為後續寫作基準。許多人試圖獨自寫完所有 hub，導致第四階段停止；重點不在數量，而在先做出足以交給 AI 工作夥伴的、文體與觀點清晰的基準範例。

完成第一個 Vault 後，隔天再打開看看。過了一夜，同一個 Vault 會以不同眼光顯現，遺漏之處和錯誤連結會浮現；它們將成為第一份 P2 議題清單，原封不動帶入下一輪。

## 延伸至姊妹文章的內容

如何運用 Librarying 建立的 Vault 寫短篇，會在《Storytelling 100》處理。Vault 的人物詞條會成為角色表，世界觀軸會成為日常片段素材的軸。

如何在翻譯工作中運用 Vault，會在《Seoyeongak》處理。翻譯註解欄、異稱欄與別名欄會成為翻譯一致性的資料。

如何從零塑造角色時參照 Vault 的世界觀軸與人物關係網，會在《New Character Build》處理；這正是第1章介紹為本書姊妹文章的那篇。如何把 Vault 的 1,500 個詞條收束為一本出版百科全書，會在《LOREBOOK》處理。

這些文章與本書在同一條管線中彼此連接。

## 面對空的 CSV

在空白 CSV 檔輸入第一列的瞬間，就是 Librarying 的開始。

`IDX`、`分類`、`關鍵字`、`說明`、`出處`。第一列填入標頭，下一列放入第一個關鍵字，第一階段抽取便開始。

那個關鍵字經過數百列，在第二階段整合中被聚為代表關鍵字，第三階段成為 Markdown 檔，第四階段寫入敘述。檔案在 Obsidian Graph View 中成為一個節點，連結接到節點上。節點累積成關係網，而這個關係網就是 IP 的世界觀地圖。

起初空白的地圖，如今已填滿 1,500 個點與數千條連線。人物在其中生活，場所存在，事件發生，裝置運作，概念流動；這個 IP 的世界已展開為一張地圖。

正如無法用一個詞解釋《Star Wars》，讀者的 IP 也無法用一個詞解釋。那個關係網現在只存在於讀者腦中。

閱讀到此結束。回到桌前，寫下空白 CSV 的第一列。IP 走到螢幕上，只能從那一列開始。
