# 第9章　第4階段敘述寫作：為 Vault 注入生命

在東方王朝世界觀[「雨師府」](https://whtdrgongmail.github.io/MEJEwiki/worlds/h9-usabu/)中查找「祈雨祭」詞條，會看到「在乾旱時祈求降雨的國家儀禮」這個定義，詳細說明則整理了儀禮的順序、條件與結果。但作家只憑這些能寫出那一幕嗎？若要寫背對龜裂稻田、第一次向乾枯天空祈雨的正午景象，乃至不知是茫然還是迫切的感覺，僅有定義與詳細說明還不足以下筆。

第4階段敘述寫作，正是寫出這些敘述的階段。

## 為何需要敘述：定義不足的理由

回想第5章做出的定義：「祈雨祭是在乾旱時祈求降雨的國家儀禮，成為考驗王之德行與正統性的舞台。」它準確、自足，在其他詞條詳細說明中被引用時可立即理解，是好的定義。

但這還不夠。想在小說或電視劇中描繪初次舉行祈雨祭的人物的作家，讀完這個定義仍不能立刻寫出場面。儀禮有什麼氛圍、帶來什麼感官經驗、此 IP 世界中的人們如何接受它、第一個奉行儀禮的人此後如何改變，全部都留在定義之外。

此時需要的是**場面化切入點**。場面化切入點不是把詞條直接轉為小說句子或分鏡的完成稿，而是定義與詳細說明抓不住的創作落點。它比事件小、比鏡頭或畫格大：不是整個「祈雨祭」事件，而是背對龜裂稻田初次望向乾空的正午；王在祈雨、群眾卻已在沉默中預感失敗的場面；祭官祝文結束後連一絲風也不起的瞬間。

這個單位在各媒體有不同名稱。網路小說中是場面或視點場面；Webtoon 中是由多個畫格組成的場面；電影與電視劇中是 scene 或 sequence；遊戲中會變成事件場景、過場動畫、任務場面，有時是關卡的核心瞬間。名稱不同，功能卻相同：把詞條概念放降為讀者、觀眾或玩家實際能經驗的瞬間。

場面化切入點有若干類型。第一是**感官型**：龜裂田地的泥土氣味、正午白光、覆塵祭壇、不發一聲的群眾，捕捉詞條的感官溫度。第二是**行為型**：祭官舉起沒有水的器皿、王跪在乾地上、百姓不看天空而看水門，這些是詞條以身體行為顯現的瞬間。第三是**關係型**：誰在誰面前執行儀禮、誰相信或懷疑它、視線的位置如何分開。

第四是**衝突型**。當祈雨祭不只是祈雨儀禮，而是考驗王之德行與正統性的舞台時，詞條不再是說明，而成為壓力。第五是**轉換型**：儀禮結束後因什麼也沒發生而不再相信天空的人，或因一絲遲來的雨而重新鞏固權力秩序的瞬間。不是每次都須填滿這五種，但重要 hub 詞條至少應有一個場面化切入點，作家才能把它轉為實際場面。

百科全書編纂學長久以來區分定義與解說。定義以邏輯敘明類別與特性，解說則講述對象實際如何運作、置於什麼脈絡、何處有趣。好的百科條目從定義開始、接著解說。Librarying 中的 `%% LLM %%`區塊就是盛放解說的地方。若定義與詳細說明是邏輯性技術，LLM 區塊便是顯示詞條在 IP 世界中是活著存在的敘述。

有了這種敘述，兩件事會改變。第一是 IP 創作者參考 Vault 時的經驗：只有定義的 Vault 是辭典，填滿 LLM 區塊的 Vault 是 IP 世界的記憶庫；翻辭典與漫步記憶庫是完全不同的經驗。第二是 AI 工作夥伴把 Vault 當脈絡使用時的品質。AI 在 Vault 參照「祈雨祭」時，只有定義便會給出一般知識層次的回答；有 LLM 區塊敘述則會以此 IP 獨有質地回答。Vault 的密度決定產出的密度。

我在遊戲中最鮮明地看見，一個個詞條的敘述匯集本身就能讓世界活起來。FromSoftware 的《艾爾登法環》與《黑暗靈魂》節省主線過場動畫，轉而在武器、防具、護符等道具的每一行說明中放入世界史片段，玩家收集碎片並在腦中重建世界。我把這作為第4階段敘述的模型：若每個詞條都如同這樣的說明文搭建，即使作家不按順序零散閱讀 Vault，碎片也會聚成活著的世界。《空洞騎士》也遵循相同原理，隨著擊敗敵人而填滿的世界內圖鑑「獵人日誌」在 168 個項目中記下敵人資訊、獵人的視線與世界歷史，使圖鑑本身成為解開世界的入口。

## 第4階段寫作的結構

第4階段寫作分四步進行。

首先**親自寫 hub 詞條。** 製作人從 100 至 150 個 hub 中親自寫五至十個代表詞條，這些敘述成為此後全部寫作的語調標準。詞條沉重的 IP 只寫五個也可能足夠；語調不易掌握的 IP 則需要寫近十個。

接著**把其餘交給 AI 工作夥伴。** 將剩餘 hub、中間詞條與 leaf 分為批次委任；批次大小與同時處理數不是固定值。詞條較輕可在一批放更多，混入主角或核心裝置等比重大的詞條便減少批次大小。

**經過五階段檢閱。** 不把委任結果原樣注入檔案，只有通過五階段檢閱的敘述才進入 LLM 區塊。

最後**更新 manifest。** manifest 是摘要建置結果的檔案；每一批完成時記下進度，工作中斷後重啟便能確認該從哪批繼續。

## 親自寫五至十個 hub：尋找聲音的過程

人親自寫的五至十個 hub，不只是品質高的敘述，而是製作人首次找到此 IP 敘述聲音的過程。

相同內容因寫作者的觀點與文體不同，會變成完全不同的文字。描述祈雨祭的敘述可以是中立記錄儀禮程序的史官聲音，也可以是從內部看初次奉行儀禮之人物的迫切與茫然的聲音，還可以是傳述這個世界古老天命信仰的說書人聲音。選擇哪種聲音不是技術決定，而是創作決定；hub 直接寫作正是在此做出決定。前面三個用來試驗聲音，寫過五個便抓到方向，到十個結束時這個 IP 的敘述聲音就完成了。

因此 hub 直接寫作是每個詞條最費心的工作，每一詞條需二十至三十分鐘。五個能建立最低限度的語調基準，接近十個則成為跨越世界觀多個軸的基準案例集。不能節省這段時間。穩固的 hub 親筆稿帶來兩件事：AI 工作夥伴以 hub 敘述為範例學習模式，剩餘委任結果品質會提高；檢閱五階段中的第四階段，也即語調一致性，會變得容易得多，因為基準愈清楚就愈容易發現偏離。

## 人物詞條的四層篇幅

第4階段寫作會依詞條比重套用不同篇幅，人物詞條尤其分為四層。

**主角**（作品核心視點與衝突軸）寫成六至八段的自由敘述，包含敘事意義與象徵、人物關係、確定外形與所屬，以及此人物在 IP 世界中扮演的角色。

**主要人物**（參與多數事件並影響敘事的人物）寫三至五段；**配角**（出現在部分事件或場面的人物）寫一至兩段；**單角**（姓名只出現一兩次的人物）寫一至兩句，只記初次出現位置與簡要角色。

為何單角只寫兩句？單角是 leaf，對 leaf 投入 hub 等量工夫會破壞平衡。Vault 內詞條的篇幅差，正顯示其在 IP 內的比重。所有詞條若寫成同樣篇幅，反而沒有任何事物被強調。

Wikipedia 的人物條目分布也遵循這四層。「莎士比亞」是超過一萬字的長篇敘述；《哈姆雷特》的主要人物「霍拉旭」有約千字的獨立文檔；在第一幕第一場短暫出現的衛兵「法蘭西斯科」甚至沒有獨立文檔，只在角色清單中有兩三行。篇幅必須不同，才能讓比重經由篇幅顯現；同樣的四層也運作於 Librarying 人物詞條中。

將這四層套用到目前處理的「雨師府」IP。以肉眼讀天、準確預測雨的例外者[[夜天子]]是主角。他的準確本身是成為大逆罪的唯一變數與敘事衝突軸，因此以六至八段寫得最深。站在同一規則另一端的雨師府大提學也有近於主角的比重，依此寫作。主要人物是直接涉及預測、水與儀禮的[[雨師府]]官員，以及掌握蓄水與水路的權門，寫三至五段。配角是只在部分場面出現的雨師府末端官員與上游村鎮鄉吏，寫一至兩段。單角則是如相信天象播種卻在乾旱中崩潰的下游村鎮百姓，只被提及一兩次的人物，以一至兩句記下初登場位置和角色。同一人物群會按比重分成四種篇幅。

此階層區分與第8章 hub/leaf 區分一併適用：既是主角又是 hub 的人物寫得最深最長，既是單角又是 leaf 的人物寫得最短。有時階層與 hub/leaf 會錯開，例如單角卻連著 IP 核心裝置、被引用很多；此時依 hub 判斷。

## 非人物詞條的結構化

非人物分類中，敘述也有需要放入結構化資訊的部分。

裝置詞條先以敘述展開裝置如何在 IP 世界中被經驗，接著設置「限制條件」區塊。整理裝置能做與不能做之事並附出處；推測與解釋放進自由敘述區，只有出處明示的規則放進限制條件。這個區塊重要，是因為裝置限制就是敘事的來源；只有知道裝置不能做什麼，才能判斷裝置出現的場面中什麼可能發生。

場所與組織詞條先以敘述展開其在 IP 世界中的角色，然後設置「構成要素」區塊，列出所屬人物與相關裝置。

事件詞條裝載的是事件在 IP 時間線的位置、參與人物、事件為 IP 世界觀帶來的變化。事件詞條特別重視因果：它為何發生、結果改變了什麼，正是此區塊的核心。

有了這些結構化區塊，後續工作才能從 Vault 準確取出資訊。若只有自由敘述而無結構，工具從文字擷取資訊時會出錯。

## 故事摘要：第4階段的核心輸入

第4階段寫作交給 AI 工作夥伴的核心輸入之一，是故事摘要。

委任詞條時應一併提供什麼？最低限度是詞條定義與詳細說明；但只給這些，AI 會在沒有此 IP 脈絡的情況下以一般知識組織敘述。只給「祈雨祭」定義，AI 會寫東亞祈雨儀禮的一般說明，卻無法知道祈雨祭在此 IP 世界觀中的意義。

故事摘要解決這個問題。它不是把原始文件縮短的摘要，而是為第4階段寫作準備的已解釋脈絡檔案。製作人重新閱讀稿件、腳本、規則書等資料，將分場劇情、人物關係與動線、核心事件時間線、象徵與母題的重複模式壓縮為一個整體。篇幅通常以原本約五分之一為目標，但核心不在節省 token，而是解釋哪個事件為何重要、哪種感覺重複、哪段關係在哪裡轉折，使其成為第4階段散文的材料。附上它之後，AI 才會在此 IP 獨有脈絡內寫作，了解祈雨祭在此 IP 中如何發生於某人物、那份經驗如何在敘事裡呈現。

要放入的內容很明確。分場劇情不可只寫「A 做了 B」，而應寫成「A 做了 B，於是 C 改變」，讓因果連結浮現。人物關係要同時寫初始狀態與最終狀態；明示 IP 中重複的感官母題，即色彩、氣味、聲音，以及象徵模式。核心事件時間線按因果而非年代排列。不放入的東西也很清楚：不要原封轉錄原文句子。放入引文不但會使篇幅超過五分之一，也會使 AI 把原句直接帶進敘述。只出現一兩次的單角描寫也因相同理由省略。摘要目的是傳遞 IP 的脈絡骨架，不是再現原本。

故事摘要做錯時，結果會在第4階段五階段檢閱中顯現。若原文摘錄過多或篇幅超過原本五分之一，AI 會把句子直接拉進敘述，LLM 區塊接近原本再現，複製的是原本文體而非 IP 獨有詞彙與質地。若過度壓縮而失去核心因果與感官母題，AI 則以一般知識補足缺口；這時第五階段的專案脈絡適合度會有更多不合格，重新要求次數增加。

故事摘要正確時，五階段檢閱通過率提高、保留詞條數減少，其品質直接影響整個第4階段的時間預算。人在第4階段前製作一次約需三十至六十分鐘，這就是第3章時間預算將其列為獨立項目的理由。

## 委任結構：批次與平行處理

不能一次把所有剩餘詞條委任給 AI 工作夥伴。一個請求可容納的脈絡有限，處理過多尤其會使檢閱標準模糊。

Librarying 的委任結構是批次處理。基準可設為一批十五個、同時處理三批左右，但數字不是絕對規則。混入 hub 或主要人物等比重大詞條時，縮減到八或十個；以 leaf 為主的輕詞條，則可增至二十個上下。同時處理也是同理：需要嚴格對齊語調的批次減至兩個，格式穩定的 leaf 批次可增至四個。

決定批次大小的標準不只處理效率，更重要的是**檢閱單位的一致性**。一批必須小到足以讓檢閱者以同一把尺一次檢查格式、連結、篇幅、語調與專案脈絡。太大時同一批前後語調會不同，太小則整體工作過度零碎。把同一世界觀軸或同一分類的詞條編在一起也是為此：把天命軸編一批、水軸編另一批，AI 工作夥伴可在批內維持該軸的詞彙與脈絡一致，人也可用同一標準檢閱。

## 實際追隨一次檢閱循環

選一個小 leaf [[堰的水門]]來看檢閱怎麼運作。在「雨師府」，水門既是分配生命的水路，也是掌水者讓一座村鎮乾枯的把手。

若 AI 草稿只把它寫成「調節水流的農耕灌溉設備」，內容沒有事實錯誤，格式、連結、篇幅與語調也可通過前四階段，但會在第五階段停下。它是任何農耕世界都成立的一般知識，缺少此 IP 的脈絡：上游權門可關閉水門，向下游敵對村鎮降下旱災；那場旱災看似天災，實則是動搖天命與王的正統性的政治。草稿說的是功能，通過稿必須說出水門的權力。

重新要求只需明確指出：不要寫一般灌溉裝置，而要寫成上游掌水者用以向下游降旱、動搖天命的武器。若三次重新要求後仍未通過，就把詞條標記為「保留」，改由人直接處理並記錄到 manifest。

## 委任後的五階段檢閱

不把 AI 工作夥伴寫出的敘述原封不動注入檔案。無論是直接寫入檔案，還是先收中間成果再由人反映，重要的是同一件事：**在通過五階段檢閱前，不確定反映到 Vault。** 這條停止線一旦崩壞，第4階段雖會變快，整個 Vault 的語調與格式卻會鬆散。

第一是格式一致性。確認 LLM 區塊結構、結構化區塊標題、欄位順序是否與格式範例一致，以格式百分之百一致為通過標準。這是在確認是否處於可自動化處理的狀態；標題或欄位順序不同，讀取檔案的後續工作便會誤讀資訊。第二是連結完整性，檢查正文的 `[[連結]]`是否都在 Vault 中實際存在、是否沒有指向自己的連結，以零個斷裂連結為通過標準。AI 寫敘述時有時會把不存在的詞條作成 `[[連結]]`，這可由自動檢查偵測。

第三是符合篇幅標準。以人物四層或 hub/leaf 篇幅標準的±30%內為通過標準，確認資源分配是否依設計進行：hub 不過短、leaf 不過長，落在範圍內即通過。第四是語調一致性，確認 hub 親筆稿建立的核心姿態是否同樣套用到其他詞條，以零個框架偏移為通過標準。自動化無法判斷，必須由人閱讀感受，是最難判斷的一階段。第五是專案脈絡適合度，檢查是否違反 profile 所定的世界觀軸、時代與文化圈詞彙，以零個脈絡違反為通過標準。AI 帶入一般知識而與 IP 設定衝突的敘述會在此被攔下；若祈雨祭描寫了此 IP 不存在的儀禮程序，便在第五階段不合格。

## 第4階段的時間預算

第4階段敘述寫作是 Librarying 最耗時的階段。親自寫五至十個 hub 需二至五小時；將其餘詞條批次委任給 AI 工作夥伴並檢閱，包含重新要求和人工修改，需三至五小時。合計五至十小時，正是第3章所分配的範圍。急著縮短時間會降低敘述品質，並在日後運營中以空缺資訊與錯誤數倍返還；應逐批仔細讓它通過。

## 一個 hub 如何建立語調基準

以「雨師府」的[[天命(天命)]]為例。定義已說明：天授王統治的權限，雨是證明，乾旱是撤回信號。第4階段則把它寫成活著的秩序：雨在，王便正當；旱來，權限便受質疑，因此談天的權限就是權力根源。若以「王朝正統性繫於一絲雨」的政治視角寫它，這個視角便決定後續詞條的詞彙與距離。它可延伸至[[雨]]、[[雨師府]]、[[王]]、[[乾旱]]與[[夜天子]]；先寫 hub，就把同一語調傳給相鄰詞條。

## 第4階段完成時

完成後，1,500 個 LLM 區塊都填入敘述；每個 .md 檔都有定義、詳細說明和敘述。讀者能在 Obsidian 追隨 `[[連結]]`，從[[天命(天命)]]走到[[雨]]、[[雨師府]]、[[乾旱]]和[[夜天子]]，以指尖漫遊 IP 世界觀並感受概念的連結與分岔。這就是活著的 Vault。

## 完成第4階段

第4階段是寫作，人的判斷比前面階段更多，一批敘述會成為下一批的標準。
