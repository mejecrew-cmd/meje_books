# 第8章　hub 与 leaf：决定词条的比重

初次打开 Vault 的图谱视图时，1,500 个点会铺满屏幕，线条在各点之间连接。仔细看会发现，点的大小并不相同。有些点汇集数十条线，有些只有两三条线抵达；这种比重差异正是第 8 章的核心：辨别哪个点是 hub、哪个是 leaf。

## 无标度网络：自然选择的结构

假设我们分析互联网中网页互相接收链接的分布。数十亿网页各自收到多少链接呢？

阿尔伯特－拉斯洛·巴拉巴西在 1999 年分析这个问题时，结果和预想大不相同。大多数页面几乎没有收到链接，Google、YouTube、Wikipedia 等少数页面却聚集了数十亿个链接。链接并非均匀分散，而是集中在少数节点。

巴拉巴西将此称为**无标度网络**（scale-free network）。这种模式不只出现在互联网；机场航线网、脑神经连接、城市道路网、学术论文引用结构中都能观察到同样现象。只要是连接自然成长的网络，几乎无一例外都会遵循这种模式：连接集中到少数 hub，其余绝大多数节点的连接较少。

Librarying 的 Vault 中也会出现同样模式。1,500 个词条里，收到最多 wikilink 的 100 到 150 个会自然浮现，这些就是 IP 的 hub；其余 1,300 到 1,400 个是连接较少的 leaf。这并非人事后任意指定，而是原本就存在于 IP 中的结构。长期喜爱某 IP 的粉丝若被问“这个 IP 最重要的是什么”，往往能马上列出十到二十项，正因为他们凭直觉知道这个结构。Vault 的 hub 工作就是用数据确认那份直觉。

完成的故事里也能看到这种分布。网络小说《全知读者视角》中，星座、化身、鬼怪不断出现，剧中剧篇幅甚至超过 3,000 话，但叙事终究回到金独子、刘众赫、韩秀英三人。引用汇集的 hub 是这少数人物，其余众多存在是填满一个场景后散去的 leaf。游戏《英雄联盟》也有 170 多名英雄，但故事实际展开的舞台集中在德玛西亚、诺克萨斯、恕瑞玛等十多个核心区域。角色可以分散成上百个 leaf，引用集中的区域 hub 却只有少数。

## 百科全书早已做了数百年这种决定

用不同方式处理连接多与连接少的项目，不是 Librarying 的发明，而是百科全书编纂学数百年来一直在做的事。

以 1768 年首次发行的《大英百科全书》为例，其中既有一行定义的条目，也有长达数十页的专题。“小提琴”可能只有两段，“音乐”却长达数十页；“伦敦”以简短定义收束，“大英帝国”则被视为长篇专题。编辑委员会花很长时间判断哪些项目值得获得长篇待遇；这项决定形成百科全书的知识层级，也左右读者如何探索知识。

数字百科全书 Wikipedia 也是如此。数千名编辑历经数十年打磨的“大韩民国”条目超过数万字，小村落条目则可能只有人口和地理信息的两三行。连接越多、重要性越高，投入的工夫就越多。Librarying 的 hub 和 leaf 区分，正是把这个传统带进 Obsidian Vault：hub 相当于 IP 的大英百科专题条目，leaf 相当于简略定义条目。

## hub 与 leaf 的定义

**hub**是其他词条频繁引用的词条；在 Obsidian 图谱视图中，许多线汇集的点就是 hub。一个 IP 的核心概念、主要人物、中心舞台、核心装置都属于此类。用一句话概括 hub 的性质：若不了解这个词条的内容，便很难理解许多其他词条。以食物和家门为题的世界观[「传授书」](https://whtdrgongmail.github.io/MEJEwiki/worlds/h7-jeonsuseo/)中，不知道“手味”，就无法理解“家臣”“传授书”“宗妇”“禁忌食材”，因此手味是 hub。

**leaf**是引用较少的边缘词条，例如只被提到一两次的配角、短暂出现的事物、作为背景的文化词汇，其他词条几乎不引用它们。这不代表 leaf 不重要。没有 leaf，Vault 会显得空洞；一个配角虽是 leaf，却可能在一个场景中决定 IP 的气氛。leaf 是构成 IP 世界密度的词汇，只是不应像对 hub 一样投入同等的写作资源。

一个 IP 大致会呈现这种分布：hub 约占 10%，即 100 到 150 个；中间层约占 20%，即 200 到 300 个；leaf 约占 70%，即 1,000 到 1,200 个。过去曾以“被引用 16 次以上”之类的阈值说明，实际工作中却不会如此简单切分。先同时计算出现频率和被引用数，做出候选表，再由人检视候选词条、确认其是否是 IP 的核心。换成篇幅，hub 要写成多个较深入的段落；中间层介于两者之间；leaf 则写成一两句到一两段。这三个层级成为第 4 阶段叙述写作的篇幅标准，并决定 90% 的写作时间分配。

这里要先澄清一点：hub/leaf 的分布是按引用比重划分的轴；第 9 章把人物词条分成主角、主要、配角、单角四层并区分篇幅，则是按叙事角色划分的另一轴。两轴标准不同，会同时套用于一个人物；例如既是主角又是 hub 的人物，就写得最深、最长。

## 自动计数与人的决定

决定 hub 的第一项工具是自动计数。第二次整合填入相关关键词列时，计算每个词条在其他词条详细说明中出现几次；计数高的词条就成为 hub 候选。

不过，引用次数排名不能原封不动成为 hub 清单。自动计数会漏掉三种情况。

第一种是**到处出现但不是核心的词条**。若一个 IP 的所有人物都在“皇帝都市梁”度过童年，梁的被引用数会非常高，但 IP 的故事不一定围绕梁运转，它更接近共同背景信息。这类词条即使计数高，也可能适合放在中间层而不是 hub。

第二种是**引用少却决定 IP 身份的词条**。某装置可能只在一部作品出现，却决定这个 IP 的世界观原理；即使被引用只有三四次，也是在众多 IP 中区分它的精髓。它在自动计数中排名低，人的眼睛却能明确看出它是 hub；只有深知 IP 的人能发现这样的词条。

最后是**各世界观轴的核心词条**。第 6 章所定的每个世界观轴中，被引用最多的词条就是该轴的 hub 候选。它在全体引用排名可能居中，却可能是特定世界观轴内最重要的词条。把全体计数和轴别计数一起看，Vault 的结构就会立体浮现。

这三种情况由人直接判断。收到自动计数结果后，人要决定“这个词条计数低仍是 hub”“这个词条计数高仍是 leaf”。

## 确定 hub 的实际过程

确认 hub 分四步进行。

第一步是**生成自动计数前 200 名清单**。分析第二次 CSV 的相关关键词列和详细说明列，计算每个词条在其他词条内被引用的次数，取出前 200 名。这 200 个中约有 100 到 150 个会留下成为实际 hub。

接着是**快速浏览清单**。快速看过前 200 个，把共同背景信息、只是常被提到的地名、像助词般使用的词标记为“不是 hub”并排除；这一步通常会去掉 50 到 100 个。

第三步是**从计数外加入 hub**。从剩余清单之外寻找“是 IP 精髓但引用少的词条”。这需要深知 IP 的人的直觉。快速滚动全部 1,500 个词条名称，把映入眼帘的加入，通常会增加 10 到 30 个。

最后是**按世界观轴交叉确认**。逐一检查各世界观轴，查看每个轴是否有合理的 hub 分布。若家臣轴有 30 个 hub、宗家轴只有 2 个，代表宗家相关工作相对做得浅。发现这种不平衡时，要判断应重新检查宗家轴的详细说明，还是视为 IP 真实比重差异的反映。

熟练制作人完成这四步约需 30 分钟到 1 小时。此处速度很重要：每个词条考虑太久，时间会几何级数增加。因此只分成“确定是 hub”“确定不是”“不确定，先当 hub”三类。犹豫的项目可在第 4 阶段写叙述时再调整。

## hub 清单产生的成果

hub 确定后会产生两项成果。

其一是**IP 核心词汇词典**。100 到 150 个 hub 的清单本身就是 IP 的核心词汇词典。先把 hub 清单交给初加入 IP 的编辑、译者或伙伴，他们不必读完 1,500 个，只读 100 个就能理解 IP 骨架，最快掌握核心。

其二是**第 4 阶段写作的顺序表**。hub 的确定决定第 4 阶段叙述写作的顺序；因为 hub 叙述是后续所有写作的语调标准，所以要从 hub 开始写。

这也自然带到第 9 章。hub 词条的叙述不交给 AI 工作伙伴，而由制作人亲自撰写，有两个理由。

**hub 叙述会成为后续全体写作的语调标准。** 制作人亲自写完 5 到 10 个 hub 后，那些叙述就成为展示此 IP 叙述质地和语调的案例集。把 1,000 个 leaf 叙述交给 AI 工作伙伴时，一并交出两三篇 hub 叙述作为“请用这种语调写”的范例。没有 hub 就没有语调标准；没有语调标准，1,000 篇叙述就会各有各的语调。

**hub 叙述会成为检阅写作品质的标准。** 亲自写了 hub 叙述的人在检阅后来委派产生的叙述时，会看“是否与 hub 的态度一致”。有标准才能检阅，没有标准的检阅只会停留在主观。

由制作人亲自写约 5 到 10 个代表 hub，再把其余交给 AI 工作伙伴，这就是第 4 阶段写作的分工原则。

## 在 Vault 图谱中看 hub 与 leaf

回到第 3 阶段骨架构建完成、初次打开 Vault 时的图谱视图。确定 hub 后，在图谱视图中为 hub 词条指定更大的点或不同颜色。设置独立标签或显示值，就能在 Obsidian 中以颜色区分 hub 和 leaf，也能在安排第 4 阶段写作顺序时直接取用清单。

图谱于是改变。起初 1,500 个点看起来大小相仿，现在 100 到 150 个大点构成网络中心，其余小点附着在大点周围。这就像以大型枢纽机场为中心、小型机场连接其间的航线图，正是 Librarying 生成的 IP 结构地图。

这张地图还可读出另一件事。按第 6 章设计的世界观轴区分颜色，就能看见哪个轴出现大量 hub。若家臣轴的 hub 多，表示这个 IP 以家臣信仰为核心基础；反之，若某轴几乎没有 hub，则该轴要么尚未充分开发，要么在此 IP 中实际比重较低。检视这个分布，可以调整第 4 阶段写作的资源分配。

IP 相关者可以一起看这张地图，共享“我们 IP 的核心是什么”。即使长期制作 IP 的团队，也未必曾一眼看见整体结构；有时经营五年的 IP 团队第一次看到地图，会意识到“原来我们的 IP 是这种结构”。Vault 图谱第一次把这个结构展示出来。

## 手握 hub 清单

hub 和 leaf 不只是篇幅问题。自动计数只是数据，决定 hub 则是解释数据的判断；两个人看同一份数据，仍可能做出不同的 hub 清单。差异就是对这个 IP 的理解深度不同。hub 的决定需要制作人的判断。

当 100 到 150 个 hub 的清单完成后，它既是这个 IP 的核心词汇词典，也是第 4 阶段写作的路线图。

第 9 章是第 4 阶段叙述写作：为只有骨架站立的 1,500 个词条 Vault 写入让 IP 世界活起来的叙述。它从制作人亲自写 5 到 10 个代表 hub 开始，接着探讨如何把剩余部分交给 AI 工作伙伴并保障质量。

第 9 章和第 10 章都会出现检查，但层级不同，先在此澄清。第 9 章的五个检阅阶段，是让一个词条或一个批次通过的工作单元检查；第 10 章的三阶段验证，则是在交接前检视所有 1,500 个都已填满的 Vault 整体。前者是词条单元，后者是 Vault 单元。
