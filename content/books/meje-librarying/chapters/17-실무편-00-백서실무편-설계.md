# 제6편. 실무편 00. 백서 실무편 설계 - 실행 양식과 기준 예시

제6편. 실무편 묶음은 본문을 더 길게 만들기 위한 장식이 아니다. 14장 본문이 라이브러링의 원리와 흐름을 설명한다면, 실무편은 그 흐름이 실제 작업에서 어떤 입력과 출력으로 남는지를 보여 주는 실행편이다.

백서라는 이름을 붙이려면 방법론의 설명만으로는 부족하다. 독자가 "그럴듯하다"에서 멈추지 않고, "이 양식을 열면 바로 시작할 수 있다"까지 가야 한다. 그래서 실무편은 산문보다 양식, 샘플, 체크리스트, 판정 기준을 우선한다.

## 실무편 구성

| 편 | 이름 | 역할 |
|---|---|---|
| A | 프로젝트 프로파일 템플릿 | 1차 추출 전 IP의 작업 조건을 고정한다. |
| B | 1차 추출 CSV 샘플 | "먼저 모으고 나중에 정리한다"는 원칙을 실제 행으로 보여 준다. |
| C | 2차 통합 CSV 샘플 | 흩어진 표기가 표제어 단위로 묶이는 모습을 보여 준다. |
| D | Vault `.md` 기준 문서 예시 | 3차 골격 빌드의 기본 파일 모양을 보여 준다. |
| E | `_vault_manifest.json` 예시 | 진행 상태, 통계, 배치, 핸드오프를 기록하는 방식을 보여 준다. |
| F | 검증 체크리스트 | 자동 단, 수동 샘플 단, 통합 빌드 단의 합격 기준을 고정한다. |
| G | 적용 전후 비교 양식 | 처리 시간, 오류, 온보딩 비용을 과장 없이 측정한다. |

## 실무편 작성 원칙

첫째, **실측과 예시를 구분한다.** 실제 프로젝트에서 나온 값은 실측이라고 적고, 설명을 위해 만든 값은 가공 예시라고 적는다. 둘을 섞으면 백서의 신뢰도가 떨어진다.

둘째, **자동화가 할 일과 사람이 판단할 일을 구분한다.** CSV 파싱, 링크 무결성, YAML 검사, 파일 수 대조는 자동화 대상이다. 동의어 묶음, 세계관축 신설, hub 확정, 번역 판단은 사람의 검수 대상이다.

셋째, **합격 기준은 판정 가능해야 한다.** "좋게 쓴다"는 기준이 아니고, `PASS / WARN / FAIL`로 판정할 수 있어야 한다. 산문 품질처럼 정량화가 어려운 항목도 표본 검수 기준과 검수자 메모를 남긴다.

넷째, **실무편은 본문을 대체하지 않는다.** 실무편은 본문을 읽은 사람이 작업에 들어갈 때 쓰는 도구다. 본문은 왜 그렇게 하는지를 설명하고, 실무편은 무엇을 채우고 무엇을 확인할지를 보여 준다.

## 예시 IP

실무편의 가공 예시는 하나의 세계, **「누비의 땅」**을 기준으로 맞춘다. 누비, 고쳐 짜기, 무문자, 씨족 장부, 회색 직조당, 붉은 실은 모두 같은 가공 IP에서 나온 표제어다. 이 통일은 예시가 흩어진 기능 샘플처럼 보이지 않고, 한 IP가 1차 CSV에서 Vault 문서와 manifest까지 이어지는 과정을 보여 주기 위한 장치다.

## 공개 백서로 전환할 때의 권장 제목

**MEJE 라이브러링 백서: 창작 IP를 Vault로 운영하는 4단계 워크플로**

이 제목은 원고의 실제 범위를 가장 잘 담는다. "세계관"보다 넓게 IP 어휘, 관계망, 번역 메타, 검증, 핸드오프를 포괄하되, 스스로를 산업 표준이나 검증 보고서로 부풀리지 않는다. 공개본의 핵심 명칭은 끝까지 **워크플로**다.
